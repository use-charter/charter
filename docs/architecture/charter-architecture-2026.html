§0 Product Brief

One-Line

Charter is a single Go binary that answers: "Is this repo safe for an AI agent to work in?"

The Problem Charter Solves

AI coding agents can read, write, execute, and commit in any repository they're given access to. Most repos were never designed with an agent in mind. The result is a set of silent, compounding failure modes that neither the developer nor the agent will catch without tooling:

Missing or malformed context files
The agent has no AGENTS.md to read, so it guesses at project conventions, makes wrong tool choices, and produces commits that violate the project's style. The developer has to undo the damage manually — every time.
Secrets in agent-visible files
An API key placed in AGENTS.md "for convenience" is now readable by every agent session, every tool call, and every MCP server the agent talks to. The agent may log it, send it upstream, or include it in a commit diff.
Unpinned MCP servers
A .mcp.json pointing to npx @acme/mcp-server@latest fetches whatever code the package registry serves at run time. A supply chain compromise gives an attacker a persistent tool inside the agent's context — undetected.
Non-reproducible environment
The agent installs dependencies, runs tests, and says "it works." The human clones the repo six months later and nothing works. No toolchain pin (mise.toml / go.mod toolchain / rust-toolchain.toml / .nvmrc…), no lockfiles, no devcontainer — the agent was flying blind on an undeclared toolchain.
No CI gating on agent workflows
Agent-produced PRs merge without any automated check that Charter itself passes. Within weeks the repo drifts: old AGENTS.md, new MCP servers added without auditing, secrets creeping back in through auto-generated config.
No single source of truth for compliance
An ISO 42001 or SOC 2 auditor asks "what can your AI agents access and what prevents them from leaking data?" The answer is a shrug. There is no artefact, no score, no policy enforcement — just a collection of ad-hoc configuration files nobody owns.

Charter makes all of this observable in a single command and fixable in under ten minutes. The score is the single artefact that answers the auditor's question.

Five Questions Charter Answers

  1. Are agent config files (AGENTS.md, CLAUDE.md, Codex config) present and correctly structured?
  2. Are secrets exposed in agent-visible files?
  3. Are MCP servers version-pinned, URLs in a trusted catalog, and remote servers declaring OAuth 2.1 auth metadata?
  4. Is the environment reproducible (language toolchain pinned: mise.toml / go.mod / rust-toolchain.toml / .nvmrc / bunfig.toml / pyproject.toml / .ruby-version / etc., lockfiles committed, devcontainer or Nix flake optional)?
  5. Are CI pipelines configured to run with agent-related workflows?

How Charter Works

Charter runs entirely offline. No data leaves your machine. No LLM is called. Every finding is produced by a deterministic rule engine that reads files already present in your repo.

01
Scan
Charter reads every file an AI agent would see — AGENTS.md, CLAUDE.md, .mcp.json, CI workflow YAML, mise.toml, devcontainer.json, lockfiles, and .gitignore. No network call. No subprocess spawned outside your repo.
02
Score
15 rules (12 core + 3 governance) emit findings tagged Blocker / High / Medium / Low. The score formula weights each severity, applies hard caps for secrets and blockers, and outputs a single integer (0–100) that CI can gate on.
03
Act
Every finding carries a rule ID, the exact file+line evidence, and a plain-English fix suggestion. Run charter fix to apply auto-fixable items after reviewing a diff. In Phase 1.5 / v1.1, charter doctor --for-agent adds TOON-format output an AI agent can act on directly.

Core Data Model

These are the canonical Go types that flow through Charter's rule engine. Every contributor and every AI agent working on Charter should understand these four abstractions before touching any domain code.

Rule
// A single deterministic check.
type Rule struct {
  ID          string    // "AE-SEC-001"
  Category    Category // CTX|SEC|MCP|CC|ENV|CI
  Severity    Severity // Blocker|High|Med|Low
  Description string
  Check       func(*Repo) []Finding
}
Finding
// One violation emitted by a Rule.
type Finding struct {
  RuleID     string     // "AE-SEC-001"
  Severity   Severity
  Confidence Confidence // High|Medium|Low — emitted alongside Severity
  File       string     // relative repo path
  Line       int        // 0 if not line-specific
  Evidence   string     // exact triggering text
  Message    string     // human explanation
  Fix        string     // plain-English suggestion
  FixAuto    bool       // charter fix can apply?
}

// Confidence is orthogonal to Severity.
// High   = deterministic pattern match (exact key prefix + correct length)
// Medium = heuristic match (high entropy, partial pattern) — output says "likely"
// Low    = structural concern, not a confirmed violation — output says "possible"
// CI blocks on Blocker+High, Blocker+Medium. Low-confidence findings are reported
// but never block — they surface in the summary for human review only.
type Confidence string
const (
  ConfidenceHigh   Confidence = "high"
  ConfidenceMedium Confidence = "medium"
  ConfidenceLow    Confidence = "low"
)
Score
// Result of a full scan.
type Score struct {
  Base       int       // max(0, 100−penalties)
  Final      int       // min(base, active_cap)
  Cap        *Cap      // nil if no cap triggered
  Findings   []Finding // active findings
  Suppressed []Finding // excluded, listed only
  Pass       bool      // Final ≥ threshold
}
Repo
// Parsed view of the scanned repo.
type Repo struct {
  Root      string            // absolute path
  Files     map[string][]byte // agent-visible only
  AgentsMD  *AgentsDoc        // parsed AGENTS.md AST
  MCPConfig *MCPConfig        // merged MCP JSON/YAML
  Suppress  []Suppression     // from .charter-suppress
  GitMeta   GitMeta           // last 20 commits, branch
}
Rule Engine Flow

At runtime: Repo is built by parsing all agent-visible files → each registered Rule.Check(repo) runs in parallel → Finding slices are merged → suppression filter applied → Score computed → output renderer selected (text / json / markdown / sarif in v1; toon added in Phase 1.5) → exit code set. The entire pipeline is synchronous from the caller's perspective; parallelism is internal to the rule runner.

v1 Rule Families — 15 Rules (12 Core + 3 Governance)

Charter v1 ships exactly 15 rules: 12 core rules across 6 categories that scan repo state and feed the Charter Score, plus 3 governance rules (AE-SUPPRESS-001/002/003) that fire on Charter's own suppression system. Every rule has a stable ID, a default severity, and a suppression mechanism. The rule set is frozen within a major version — no silent additions, no severity changes without a semver bump.

CLI at a Glance

Charter exposes four commands. doctor is the only one you need to know. The rest exist for automation and remediation workflows.

charter doctor — primary scan
--format text|json|markdown|sarif Phase 1.5: --for-agent --rule AE-SEC-001 --threshold 80 --quiet --no-color
Runs all 15 rules (12 core + 3 governance). Exit code 0 = score ≥ threshold (default 80). Exit 1 = below threshold. Exit 2 = scan error. V1 outputs are text, json, markdown, and sarif; Phase 1.5 adds --for-agent and TOON-format structured output for AI agent consumption.
charter fix — apply auto-fixes
--dry-run --rule AE-CTX-001 --yes
Always shows a unified diff before writing. Never deletes files. Never auto-fixes secret-related rules — those require manual remediation by design (Commitment #4).
charter report — structured output for tooling
--format sarif --format spdx --out report.sarif
Emits SARIF 2.1.0 for GitHub Code Scanning, or SPDX 3.0.1 for SBOM workflows. Designed to be piped into CI artifact upload steps.
charter suppress — manage false positives
AE-CC-001 --reason "legacy repo" --expires 90d
Writes a .charter-suppress.yml entry. Suppressed findings are excluded from the score calculation and listed separately in output. Expiry dates are enforced — expired suppressions re-surface as findings.

The Ten Commitments

These are not preferences. They are the principles Charter will be held to at every build decision.

#Commitment
1Never send data anywhere without explicit opt-in.
2Never call an LLM — all findings are deterministic.
3Never delete a user file.
4Never apply a fix without showing a diff first.
5Every finding has a rule ID, evidence, and fix suggestion.
6Every release is signed (cosign) with SLSA Level 2 provenance.
7The score formula is public and unchanging within a major version.
8Always stays cross-vendor: works with Claude Code, Codex, Cursor, Windsurf, OpenCode, Copilot, Gemini.
9Never print raw secret values in any output. Never auto-fix secrets destructively. Keep false-positive rate below a measured threshold — enforced by zero-FP assertions on all clean-repo fixtures.
10The CLI is free forever. No feature is paywalled in the core scan.

§1.5 Distribution — The First-User Problem

The question this doc doesn't ask

Every other section answers "what does Charter check?" and "how is it built?" This section answers the only question that determines whether anyone uses it: why would a developer install a new CI tool today, before they have felt the pain of a misconfigured agent repo acutely? The pain Charter solves is diffuse and delayed — a missing AGENTS.md doesn't cause a loud failure, it produces subtly worse agent output over months. Tools that solve diffuse pain need either a forcing event or a trusted distribution channel. This section defines Charter's answer to both.

The Install Trigger Problem

Most CI tools get installed in one of three ways: (1) a colleague recommends it after a production incident, (2) it ships bundled inside a tool the developer already uses, or (3) it appears in a curated list (awesome-list, framework docs, security baseline) that the developer trusts. Charter cannot wait for incident-driven adoption — agent-config problems don't produce loud incidents. The bundle and list strategies are the actionable levers.

Bundle strategy — highest leverage
Get Charter referenced in the setup documentation of agent tools the developer already chose: Claude Code quickstart, Cursor docs, Windsurf onboarding. "Run charter doctor to verify your agent setup" is one sentence. If Anthropic links it, Charter installs in the same session the developer installs Claude Code.
List strategy — compounding trust
Get Charter into: awesome-mcp-servers, OWASP MCP Top 10 tooling recommendations, OpenSSF best-practices checklist, and GitHub's Actions marketplace curated security category. These are discovery surfaces developers visit when they are actively looking for security tooling — the highest-intent moment possible.
SARIF strategy — invisible distribution

§1.7 Phase 1 Exit Criteria — What "Validated" Means

Define the answer before you start building

The most common OSS tool failure mode is not a bad tool — it's ending Phase 1 without a clear answer on whether to proceed. Stars and install counts are noise. The signals below are the ones that are hard to manufacture and easy to interpret. Charter Phase 1 is complete when at least 3 of these 4 signals are true. If after Phase 1 milestones are shipped (M1.1–M1.6) fewer than 3 signals are green, stop and diagnose before investing in Phase 2 backend work.

Signal 1 — Organic CI Adoption
≥ 5 repos running charter doctor in CI that you didn't personally tell to install it
Check: search GitHub for uses: use-charter/charter-action excluding your own org. Each organic install is a real developer who decided Charter was worth adding to their pipeline. This is the hardest signal to fake and the most meaningful one. GitHub Action installs from the marketplace page also count if the install happened without direct outreach.
Signal 2 — Stranger Issues
≥ 3 GitHub issues filed by people you don't know personally, with real repo context
A stranger filing "AE-MCP-001 fires incorrectly on our monorepo because…" is proof the tool is in active use and real enough to cause friction worth reporting. Issues from acquaintances, colleagues, or people you directly asked don't count. The stranger issues tell you both that people are using it and that they trust it enough to invest time in improving it.
Signal 3 — Unprompted Mentions
≥ 1 blog post, HN comment, or social post mentioning Charter that you didn't write or prompt
Someone thought Charter was interesting enough to tell their network about without being asked. Set up a Google Alert and a GitHub search notification for "charter doctor" and "use-charter". A single genuine third-party mention — even a short one — is a stronger signal than 200 stars from a well-timed HN post. The mention means Charter solved something real for them.
Signal 4 — Community Self-Help
≥ 1 GitHub Discussion or community thread where someone answers another person's Charter question — without you being involved
When users help each other without the founder in the thread, you have crossed from "interesting project" to "community." Set up GitHub Discussions from day one. Don't answer every question immediately — give the community 24 hours to respond first. The first time a stranger helps another stranger understand a Charter rule is the moment Phase 1 has proven something real.

If Validation Fails — The Honest Diagnosis

If after M1.6 ships (the final Phase 1 milestone — MCP catalog live, full rule set locked, FP validation gate passed) and 60 days have passed, fewer than 3 signals are green, do not proceed to Phase 2. Instead:

Failure patternLikely diagnosisAction
Many stars, zero organic CI installs Interesting to read about, not useful enough to add to CI. The rules may be too noisy or the fix instructions too vague. Talk to 5 developers who starred it but didn't install the Action. Ask why. Fix the top complaint.
Some installs, zero stranger issues People installed it, it passed on their repo, they moved on. One-time setup trap is real. Accelerate the static MCP catalog so new findings fire on repos that previously passed. If catalog alone doesn't create engagement, the recurring-value model is broken.
Issues exist but only from acquaintances Social proof without real market signal. You may be in an echo chamber. Post to communities where you have no existing relationships — a Go-specific forum, a security Slack you're not already in. Measure cold reception.
All signals green but no GitHub Sponsors Validated adoption, unclear willingness to pay. Still proceed to Phase 2, but start with the cheapest possible paid feature and test conversion early. Proceed. The absence of sponsors is not a kill signal — many successful OSS tools take 12–18 months to find the right monetization model. Adoption is the prerequisite.

The Doctrine

Build first. Validate through usage. Business proof follows product proof.

Charter follows the devtool wedge playbook: taste and execution come first, usage signal second, business proof third. Pre-build interviews and LOIs are how enterprise SaaS companies validate — not how developer tools earn trust. The tool earns trust by being undeniably useful on the first charter doctor. That's the only pre-validation that matters.

Stage Signal What It Proves

Founder Dogfooding Before v1 Ships

T1.1.1 Repository Resolver ⚡ AI
Input

Current working directory (or --path flag); environment: OS, GOPATH, HOME

Output

RepoContext struct: git root, branch, commit SHA, monorepo flag, workspace name, package manager, platform, ignore rules

Subtasks
  • Walk up from cwd to find .git directory; handle git worktrees (detect .git as file pointing to worktree)
  • Detect monorepo root by presence of pnpm-workspace.yaml, turbo.json, root-level packages/ dir, or go.work
  • Detect package manager: check for bun.lock, pnpm-lock.yaml, yarn.lock, package-lock.json, go.mod, Cargo.toml, pyproject.toml in priority order
  • Read .gitignore, .git/info/exclude, and global gitignore; compose into single IgnoreRules matcher
  • Normalize all paths to forward-slash representation regardless of OS (Windows path safety)
  • Expose RepoContext.Platform: windows | darwin | linux
Acceptance Criteria
As a developer running charter doctor, I want the CLI to reliably resolve the target repository path from flags, environment variables, and current working directory, so that Charter works correctly in local, CI, GitHub Actions, and pre-commit hook contexts without manual path configuration.
Given a developer runs charter doctor from inside a git repository
When no --path flag is passed
Then Charter resolves the repo root by walking up from CWD to the .git directory and uses that as the scan target
Happy Path
  • charter doctor --path /abs/path/to/repo uses the explicit path; overrides CWD resolution
  • CHARTER_REPO_PATH env var sets the default path; --path flag takes precedence over the env var
  • Running from a subdirectory (e.g., src/): Charter walks up to repo root correctly
  • Running in GitHub Actions: GITHUB_WORKSPACE env var is detected and used as fallback when CWD is not a git repo
  • Path resolution uses filepath.EvalSymlinks to canonicalize symlinked paths; no double-scan of the same inode
  • Resolved path is printed at debug level (--verbose): debug: resolved repo root → /home/user/myrepo
  • Unit tests cover: explicit flag, env var, CWD walk-up, symlink, and GitHub Actions fallback cases
Error Handling
  • If the resolved path is not a git repository (no .git dir/file), Charter exits with code 2 and message: error: not a git repository: /path/given
  • If the path does not exist on disk, Charter exits with code 2: error: path does not exist: /path/given
  • If GITHUB_WORKSPACE is set but points to a non-existent path, Charter logs a warning and falls back to CWD resolution
  • If CWD walk-up reaches filesystem root without finding .git, Charter exits with code 2: error: could not find git repository root from: /current/dir
Edge Cases
  • Worktree repos (git worktree add): .git is a file (not directory) pointing to the main worktree; Charter handles both cases
  • Bare git repositories: Charter detects bare repos and exits with a descriptive error — bare repos are not valid scan targets
  • Path with spaces or Unicode characters: resolved correctly on all platforms
  • Monorepo with nested .git directories (submodules): Charter scans only the outermost repo root; submodule paths are not recursed unless --include-submodules is passed (Phase 2)
Out of Scope
  • Remote repository scanning via URL (e.g., charter doctor github.com/org/repo) — Phase 2 feature
  • Auto-detecting monorepo workspace packages and scanning each independently
T1.1.2 File Inventory Scanner ⚡ AI
Input

RepoContext (root path + ignore rules)

Output

FileInventory: categorized map of interesting files found, total file count, scan stats

Subtasks
  • Parallel traversal using bounded goroutine worker pool (default 16, configurable via CHARTER_WORKERS)
  • Skip directories: .git, node_modules, vendor, dist, build, .next, target, .cache
  • Skip binary files (detect by null bytes in first 512 bytes) and files >1MB
  • Apply IgnoreRules from RepoContext to each path
  • Classify interesting files into categories: agent-instructions, agent-settings, mcp-config, env-files, toolchain, ci-workflows, lockfiles
  • Compute mtime + size hash per file for incremental cache invalidation
  • Persist cache to .charter/cache/inventory.json (gitignored)
Acceptance Criteria
As a developer, I want Charter to walk the repository tree and build a normalized file inventory that respects .gitignore, so that rule evaluations operate on a complete and accurate file set without scanning vendored, generated, or ignored files.
Given a repository with a populated .gitignore
When charter doctor runs the file inventory step
Then the inventory contains all tracked and untracked-but-non-ignored files; files matching .gitignore patterns are excluded
Happy Path
  • File walk uses go-gitignore (or equivalent) to evaluate .gitignore at every directory level — nested .gitignore files are respected
  • Inventory is built as a slice of FileEntry{Path, Ext, Size, IsDir} structs — no file content is read during inventory phase
  • Hidden files (dotfiles) are included unless explicitly ignored; .github/, .cursor/, .claude/, .vscode/ directories are always included
  • Symlinks are followed once; cycles are detected and skipped with a debug-level warning
  • Binary files are identified by extension/magic bytes and tagged IsBinary: true; rule evaluations skip binary content
  • File walk completes in ≤ 500 ms on a 50,000-file repo (benchmarked in T1.5.3)
  • Inventory is immutable after construction — rules receive a read-only view; no rule can mutate the inventory
  • Test fixtures include: a repo with nested .gitignore, a repo with vendor/, a repo with node_modules/, and a bare repo (error case)
Error Handling
  • If the repo root is not readable (permissions error), Charter exits with code 3 and the OS error message
  • If a symlink target does not exist (dangling symlink), it is logged at debug level and skipped — not a hard error
  • If a file is larger than 10 MB, it is tagged IsLarge: true and its content is not loaded for rule evaluation; a LOW finding notes the skipped file
Edge Cases
  • Repo with vendor/ directory: vendored files are excluded by default (treated as if gitignored); --include-vendor flag re-includes them (Phase 2)
  • Repo with .gitmodules: submodule directories show up in the inventory as IsSubmodule: true; their contents are not walked in v1
  • Empty repository (zero files beyond .git/): Charter completes with zero findings and a score of 0 (all rules fail due to missing files) — not a crash
Out of Scope
  • Content-addressable deduplication of identical files
  • Incremental inventory (diff from last scan cache) — planned for Phase 2 performance optimization
  • Network filesystem (NFS) support — not tested in v1

Architectural Contracts — CLI Intelligence Layer

Stable Finding Fingerprints

Every finding must have a deterministic ID stable across runs. Without this, suppressions break on re-scan, PR comments duplicate instead of update, and Cloud dashboards can't distinguish new findings from recurring ones.

// FindingFingerprint: stable hex ID, normalized to prevent path-alias churn.
func FindingFingerprint(ruleID, normalizedPath, evidenceAnchor, configKey string) string {
    h := sha256.New()
    fmt.Fprintf(h, "%s\x00%s\x00%s\x00%s", ruleID, normalizedPath, evidenceAnchor, configKey)
    return hex.EncodeToString(h.Sum(nil))[:16]
}
Finding fingerprints are stable across re-runs for the same logical finding. Suppression files reference fingerprints, not line numbers. A finding whose location moves but whose rule + evidence anchor are unchanged keeps the same fingerprint.

Baseline Mode + PolicyDecision Type

Baseline mode is the adoption mechanism for imperfect repos. The PolicyDecision type is the shared model for v1 suppressions, Phase 2 exceptions, audit logs, and compliance exports — defined once, extended never.

type PolicyDecision struct {
    ID        string      // stable UUID
    FindingID string      // stable FindingFingerprint
    RuleID    string
    Decision  string      // "suppress" | "accept" | "escalate" | "fix"
    Reason    string      // required for suppress/accept
    Actor     string      // git author or Cloud user ID
    ExpiresAt *time.Time  // nil = permanent
    Source    string      // "inline" | "charter.yaml" | "cloud"
    CreatedAt time.Time
}
Suppression rules (v1, enforced): (1) Reason is required — a suppression without a human-readable reason string is a MEDIUM finding, not a lint warning. The inline syntax is # charter:ignore AE-SEC-001 reason="test fixture — not a real credential". A bare # charter:ignore AE-SEC-001 emits AE-SUPPRESS-001 MEDIUM: suppression missing required reason. (2) ExpiresAt defaults to 90 days from creation. There is no self-serve permanent suppression in v1 — ExpiresAt: permanent is reserved for Phase 2 Cloud and requires an org-level approver and a second mandatory field: ApprovedBy (GitHub handle of the approver who reviewed the risk). A suppression file containing permanent without a non-empty ApprovedBy emits AE-SUPPRESS-002 HIGH: permanent suppression requires approver and is treated as a 90-day expiry instead. An expired suppression re-activates the original finding on the next scan. (3) Charter tracks the suppression rate per repo: if suppressed findings exceed 30% of total findings in a scan, Charter emits AE-SUPPRESS-003 MEDIUM: high suppression rate — review suppression log. Charter's v1 output surfaces are: text (human terminal), JSON (automation/API), Markdown (GitHub PR comment), and SARIF (GitHub code scanning). Phase 1.5 / v1.1 adds TOON (agent-oriented structured output), --format json-compact, and --for-agent. All project from the same canonical report object.

What TOON Is

TOON is a compact, human-readable encoding of JSON data designed for LLM prompts. It uses indentation, fewer braces, tabular arrays, declared lengths, and field headers to reduce token usage while staying readable. In its mixed-structure benchmark, TOON achieves 76.4% LLM retrieval accuracy vs. JSON's 75.0% while using 39.9% fewer tokens than formatted JSON. Charter findings are repeated structured records — exactly the shape where TOON's tabular encoding is most useful.

Honest framing: TOON benchmarks are calculated against formatted (pretty-printed) JSON. In some semi-uniform/nested cases TOON is +14.7% vs. compact JSON. Do not market this as "always fewer tokens." Market it as: agent-readable, compact, structured output for LLM workflows.

Go Implementation Status

The official TOON Go implementation is listed as in development on the TOON spec page. TypeScript, Python, Rust, and Java implementations are stable. Since Charter is a Go single binary, do not depend on the external Go package for v1. Instead, ship a deliberately-limited internal encoder: render/toon.go — encodes Charter's own ScanReport schema only, with golden tests and round-trip tests. Add the official Go library when stable.

Renderer Projection Model

One redaction pass produces the canonical redacted report. Then N stateless renderers project from it — no renderer reads another's output or accesses the filesystem.

// Render pipeline: one redaction pass, then N stateless projections.

charter explain <rule-id> and charter explain <finding-id> are natural additions — single-finding projections with full human-readable context.

Renderers are stateless projections. They receive a fully redacted CharterReport and write output. No renderer may read from another renderer, call the filesystem, or make redaction decisions. MustNotContainSecret is asserted on every renderer output in CI.

Schema-Aware Redaction Layer + Typed Guardrails

Regex-only redaction misses values in semantically sensitive keys: headers.Authorization, env.GITHUB_TOKEN, args["--api-key"]. The redaction layer combines schema knowledge with regex and entropy detection. Go types make unsafe writes unrepresentable at compile time.

// internal/redact — the single redaction boundary.
type Layer struct {
    RegexDetectors    []regex.Detector      // generic secret patterns
    SchemaFields      []schema.SensitiveKey // config field semantics
    EntropyDetectors  []entropy.Detector    // high-entropy string heuristics
    ProviderTokens    []provider.Fingerprint // GitHub PAT, AWS key shapes
    Allowlist         []allowlist.Pattern   // known-harmless examples
}

// Type-level guardrail — unsafe by default, safe only after explicit conversion
type RawEvidence     string   // from scanner; never passed to renderers or Cloud
type RedactedEvidence string   // after redact.Apply(); safe for output
type SafeOutput       string   // after renderer; safe for write to io.Writer or Cloud

// Only redact package can produce RedactedEvidence from RawEvidence
func Redact(e RawEvidence) RedactedEvidence { ... }

// Renderers accept only RedactedReport — compile-time safety
func Render(r *RedactedReport, w io.Writer) error { ... }
Redaction runs once, before any renderer, in a single isolated layer. Raw evidence values are never passed to fmt.Println, log.Printf, or any output writer outside internal/redact. Panic logs, debug traces, and test failure output pass through the same redaction boundary. Lint rule: no direct call to output functions outside internal/safeprint and internal/redact.

§7 Security Model

Non-Negotiable Invariants (Enforced Across All Phases)

Absolute Rules

These are not configurable. They are encoded as test assertions that run on every CI build.

  • Raw secret values never appear in any output. Evidence.Snippet for Secrets category findings is always [REDACTED:len=N]. Enforced by MustNotContainSecret test running on all renderer output.
  • No data upload without explicit opt-in. Phase 1 and Phase 2 are entirely local. Charter Cloud sync requires a charter sync --confirm command with explicit acknowledgment.
  • No model calls in the CLI. All findings are deterministic rule evaluations. This is a trust feature, not a limitation.
  • No MCP server execution in Phase 1. Static parsing only. Dynamic inspection in Phase 2 is gated behind charter mcp inspect --allow-execution explicit flag.
  • Generated fixes require confirmation. Dry-run is default. --yes must be explicit and is documented as CI-only usage.
  • Never delete user files. Fix operations create/patch/append only.
