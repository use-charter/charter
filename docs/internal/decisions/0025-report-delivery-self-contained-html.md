# ADR-0025 Report Delivery — Self-Contained Single-File HTML

- Status: Accepted
- Context: `charter report` is one of the six core v1.0 commands named in the canonical architecture, but it is **unbuilt**, and its only documented use (`--format spdx`, an SBOM) was deferred to Phase 1.5 — so `report` currently has no shipped purpose. Charter needs a shareable, human-grade artifact of a scan: something a developer can open, read, and hand to a teammate or attach to a PR, beyond the terminal and the SARIF that already flows to GitHub Code Scanning. The original idea was a `localhost:PORT` viewer; the question is the safest, most offline-first-aligned way to deliver a report, validated against the industry. Grounding (fetched 2026-06-02, ADR-0006): the established pattern for portable reports is a **self-contained single HTML file** with data + CSS + JS **inlined** — [Lighthouse](https://github.com/GoogleChrome/lighthouse/blob/master/report/README.md) (standalone HTML, client-side renderer), Go's own `go tool cover -html`, `go-better-html-coverage`, [Bun standalone-html](https://bun.com/docs/bundler/standalone-html), and `vite-plugin-singlefile` all ship one portable file that opens by double-click and works fully offline. The decisive technical fact: a `file://` page **cannot `fetch()` a sibling JSON** (browser blocks `file://` XHR), so the data must be inlined into the document — exactly what Lighthouse does. Local-server viewers (`go tool pprof -http`, `playwright show-report`, `vitest --ui`) exist only because those reports need **dynamic** features a static file can't serve (e.g. Playwright traces explicitly fail over `file://`); Charter's report is **static data** (findings, scores, scorecard) and needs no server. The localhost-server path also carries a real, current security surface: the **"0.0.0.0 Day"** class ([Oligo, disclosed Aug 2024](https://www.oligo.security/blog/0-0-0-0-day-exploiting-localhost-apis-from-the-browser)) — public sites reaching services bound to `0.0.0.0` on macOS/Linux, up to RCE — and **DNS rebinding** (mitigated only by `Host`-header allowlisting). A **hosted** `use-charter.dev/reports/###` is the opposite of Charter's offline-first promise: it would egress findings about a customer's private repository, the precise data peer scanners build brokers and "no-upload engines" to *avoid* moving (Snyk stores only pointers + offers a Broker/local engine; Semgrep emphasizes local/CI scanning with a network broker — the market direction is toward locality). Visual direction follows the committed design references at `docs/internal/designs/*.html` and ADR-0024's terminal token system, so the report is one visual family with the TUI.
- Decision: `charter report` ships as the generator of a **self-contained single-file HTML report**, and that is its v1.0 purpose.
  1. **`charter report --format html` (default)** renders one `.html` file with all data (the scan result as an inlined JSON payload), CSS, and JS embedded — **no external assets, no network, opens from `file://`**. Default output path `charter-report.html`; `--out <file>` overrides. `--open` opens it in the default browser (off by default — writing the file and printing its path is the default, honoring the no-surprise-side-effects posture). `charter report` also supports `--format markdown` and `--format json` (sharing the existing renderers); **`--format spdx` (SBOM) stays deferred to Phase 1.5**.
  2. **No local server in v1.** A `charter report --serve` viewer is **deferred** (Phase 1.5). When/if built it MUST follow the full hardening checklist recorded here: bind `127.0.0.1` only (**never** `0.0.0.0`), validate the `Host` header against a loopback allowlist (DNS-rebinding defense), gate access with a random per-session URL token, auto-open then auto-shut-down on `Ctrl-C`/idle, and expose no write endpoints. It is deferred because for a static report a server adds **no capability** over the single file (which `--open` already launches) while adding that attack surface.
  3. **Hosted `use-charter.dev/reports/###` is rejected for the OSS core.** It contradicts offline-first (Commitment #1) and creates a standing liability (custody of findings about private code). It is reserved, if ever, for the paid hosted-service tier as an **explicit opt-in upload** with a published data-handling posture (pointers-not-source, retention, regions) — and even then a paste/drag "viewer" is preferable to auto-upload. The free, zero-privacy-cost hosted surface that already exists is **SARIF → the customer's own GitHub Code Scanning tab**.
  4. **Audience & content.** The report is for **one human reading one repo's one scan** — the developer/owner who ran it (or pulled it as a CI artifact), and a teammate/lead who receives the file and must understand it **without the CLI installed**. It is *not* for agents/CI (they use `json`/`sarif`) nor a multi-repo fleet dashboard (that is the future hosted/paid tier). Content = a complete, self-contained snapshot of `doctor.Result`: provenance header (repo, git ref, version, timestamp, threshold/profile, duration); score hero with any active **cap** explained; the per-category scorecard; every active finding (severity, `path:line`, **redacted** evidence, remediation, auto-fix hint, OWASP mapping, embedded `charter explain` rationale + `helpUri`); suppressions + governance flags; and summary counts. Out of scope (v1): run history/trends/diffs, multi-repo aggregation, live re-scan or fix-*application* (the report shows the `charter fix` hint; applying is the CLI's job), and raw-source embedding (pointers only). Interactivity stays client-side and inlined: filter by severity/category, text search, expand/collapse, copy `path:line`, light/dark — zero network.
  5. **Design, fonts & accessibility.** The report takes **inspiration (not a copy)** from `docs/internal/designs/charter-html-report.html` and shares the Charter token system (`docs/internal/designs/DESIGN-TOKENS.md`), elevated using the installed `.agents/skills/` design set (full map in `DESIGN-TOKENS.md`) — `frontend-design` (lead), `web-design-guidelines` (the WCAG 2.2 AA gate), `design-system-patterns` + `visual-design-foundations` (tokens / type / color), `interaction-design` (microinteractions / feedback), `design-taste-frontend`, `high-end-visual-design`, and `ui-ux-pro-max` — respecting the existing token system, with a screenshot-verify pass at build. Typography (canonical, `DESIGN-TOKENS.md`): **Ruda** (`--font-site`/`--font-audit`) for body / headings (700/800) / finding titles (700) / table text (500) / badges (700 uppercase); **Atkinson Hyperlegible Mono** (`--font-code`) as the **primary** mono for all code / CLI / terminal / config / inline code; **IBM Plex Mono** (`--font-meta`) *optional*, only for the formal metadata layer (rule IDs, rule paths, file paths, confidence) — do **not** default code/rules to it; **Share Tech** (`--font-accent`) for status accents only — all SIL OFL 1.1 and **embedded as Latin-subset `woff2` inline** (no CDN, no fetch — ADR-0001 / zero external assets), each stack ending in a system-font fallback. The `✦ charter` wordmark uses Ruda 700/800 (Space Grotesk dropped — too common). The score + category scorecard are the hero; severity always carries a **text label** (never color-only); light/dark via `prefers-color-scheme`; **WCAG 2.2 AA** — at minimum the criteria the reference annotates: 2.1.1 keyboard operable, 2.4.7 focus-visible, 2.5.5/2.5.8 target size, 3.3.2 labels, 4.1.2 name/role/value (`role=progressbar` score, `aria-expanded` cards, `aria-pressed` filters), 1.3.1 semantic structure. Deterministic ordering (a generation timestamp is allowed — the report is not a committed golden).
- Consequences: new `internal/render/html` package (consistent with `internal/render/{json,markdown,sarif}`), with CSS/JS/template assets embedded via `go:embed` and unit-tested for self-containment (assert the output references no `http(s)://` asset and no external `src`/`href`). It reuses the finding/category model and the `charter explain` rationale text (ADR-0024) for the report's per-rule cards, so the explanation surface is authored once. `cmd/charter` gains the `report` command (`--format html|markdown|json`, `--out`, `--open`). The score formula (ADR-0008), SARIF golden, and `json` DTO are untouched. Commitments preserved: offline-first (#1), no network (#7), no silent mutation (the report is write-only to the chosen path, never overwriting source). Architecture is amended: `charter report`'s documented purpose becomes the HTML report (html leads; spdx labeled Phase 1.5), and the §1.8 gallery's `report` screen is aligned. Deferred and recorded: `--serve` (with the hardening checklist above) and `--format spdx`. Related: ADR-0001 (offline-first core — the load-bearing reason hosted is rejected), ADR-0024 (TUI/terminal output — shared tokens + explain model), ADR-0014 (rule catalog), ADR-0008 (score formula, unchanged).
