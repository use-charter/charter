# ADR-0009 Structured Finding Location

- Status: Accepted
- Context: Charter is a SARIF 2.1.0 scanner whose v1 surface (`--format sarif`, the GitHub Action upload, `charter fix`, and the documented `text|json|markdown` output) all reference a file path and line per finding (for example `AGENTS.md:14`). The current `findings.Finding` carries only free-form `Evidence []string`, so no consumer can obtain a structured location, and the rule layer does not capture line numbers at all. This blocks M1.5 (SARIF + Action) and M1.4 (`charter fix` needs to know where to act).
- Decision: Add `Locations []Location` to `findings.Finding`, where `Location` is `{ Path string; Line int }` with repo-relative forward-slash paths and a 1-based `Line` (`0` means file-level / no specific line). Serialize it in the JSON contract as a `locations` array of `{ "path", "line" }` objects shaped to map directly onto SARIF `physicalLocation` (`artifactLocation.uri` + `region.startLine`). Rules populate a location wherever a physical site exists; absence findings (no context file, no toolchain file) carry an empty array. Scoring stays location-agnostic. An array is chosen over a single location so the contract matches SARIF natively and never needs a breaking migration when a rule reports multiple sites.
- Consequences: `schemas/` gains a versioned finding/result schema that includes `locations`; the secrets detector path and `AE-SEC-001`/`AE-SEC-002` must capture and emit line numbers; the text and JSON renderers display `path:line`; `evidence` strings are retained for human-readable detail and are no longer the carrier of location data; the SARIF renderer (M1.5) and the fix engine (M1.4) build on this contract; the documented output examples and the affected `AE-*` specs are updated to match. The JSON change is additive (new optional field) and lands before any external consumer depends on the v1 schema.
