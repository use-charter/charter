# ADR-0014 SARIF Output & Rule Catalog

- Status: Accepted
- Context: Charter is positioned (architecture §0, §1.8) as "a single-binary, offline-first SARIF 2.1.0 scanner," and the GitHub Action (M1.5) uploads SARIF to GitHub Code Scanning so findings appear in the Security tab — the primary Phase 1 distribution surface and validation Signal 1. The data model is already SARIF-shaped: ADR-0009 gave findings structured `path:line` `Locations` (mapping to `result.locations[].physicalLocation`), and ADR-0013 added the suppression `Source` (`inSource`/`external`, mapping to `suppression.kind`) and the `Informational` flag (mapping to `result.kind: "informational"`). Two things are missing: a SARIF renderer, and per-rule metadata for `tool.driver.rules[]` (findings carry `RuleID`/`Severity`/`Category`/`Summary` but no rule name or help URI). Verified live (2026-06-01) against GitHub's code-scanning docs: an uploaded file must be **SARIF 2.1.0**, and Code Scanning specifically uses `tool.driver` (with a `rules[]` array of `reportingDescriptor`), `results[]`, and `partialFingerprints` — it reads only `primaryLocationLineHash` and backfills it from source if absent. `owenrumney/go-sarif` v3.3.0 exists but adds a dependency Charter would itself flag.
- Decision:
  1. **Hand-roll `internal/render/sarif`** — typed structs + `encoding/json`, no SARIF library, mirroring the existing hand-rolled `render/json` and `render/markdown` packages. The Code Scanning subset is ~6 object types; hand-rolling keeps Charter dependency-free (on-brand for a supply-chain scanner), deterministic (we own key order via struct fields), and is validated by golden + structural tests. `Render(doctor.Result) ([]byte, error)` emits SARIF **2.1.0** (the version GitHub ingests; not the emerging 2.2).
  2. **New `internal/rules/catalog` registry** — a static table for all 15 v1 rules: `{ID, Name, Category, ShortDescription, HelpURI}`. It is the single source of rule metadata for SARIF `tool.driver.rules[]` today and `charter explain` / `--rule` later. `HelpURI` is `https://use-charter.dev/rules/<ID>`. A **spec-sync test** asserts the catalog's ID set exactly equals the `docs/internal/specs/AE-*.md` files, so the catalog cannot drift from the behavioral contracts. The catalog carries **no severity** — a result's `level` and a rule's `defaultConfiguration.level` are both derived from the actual finding severity (single source of truth; no severity duplication).
  3. **SARIF mapping** — `runs[0]` sets `automationDetails.id: "charter"` (so Charter's results group as one logical analysis and re-runs replace cleanly without colliding with other tools). `tool.driver` = `{name: "Charter", informationUri: "https://use-charter.dev", version: version.Version(), rules: [...]}`; `rules[]` is built from the catalog for the distinct rules that produced results (active + suppressed), deduplicated and sorted by ID, each with `shortDescription`, `helpUri`, `defaultConfiguration.level`, and `properties.{category, severity}`. Non-informational rules also get `properties.tags: ["security"]` and `properties.security-severity` (Blocker `9.5` / High `7.5` / Medium `5.0` / Low `2.0`) so GitHub classifies them as security alerts and ranks/gates by severity; informational rules omit both. `results[]` carries `ruleId`, `ruleIndex`, `level`, `message.text` (the finding `Summary`), `locations[]` (`artifactLocation.uri` = repo-relative forward-slash path, `region.startLine` when line > 0), and `partialFingerprints`. Suppressed findings are emitted as results with a `suppressions[]` array (`kind` from ADR-0013's source); informational findings set `kind: "informational"`. **Severity → level:** Blocker/High → `error`, Medium → `warning`, Low → `note`. Per the SARIF consistency rule, every result carries a `suppressions` array (empty for active).
  4. **partialFingerprints** — Charter emits `primaryLocationLineHash` itself: a hex `sha256` of `ruleId\x00path\x00line` (the finding's primary location; `line` 0 for file-level), or `sha256(ruleId)` when there is no location. This is **position-based and computed purely from the finding** — no source-file I/O — keeping the renderer a pure function (like `render/json`) and never failing on fingerprinting, while giving stable cross-run alert identity even for non-Action uploads (the code-scanning API). GitHub treats `primaryLocationLineHash` as an opaque match key, so a deterministic position hash is sufficient; content-based, line-shift-resilient hashing is a deferred enhancement.
  5. **CLI surface** — `--format sarif` joins `text|json|markdown`; a new `--out <file>` flag writes the rendered output to a file (with a brief summary line to stderr) instead of stdout, defaulting to stdout when absent. Exit-code semantics are unchanged (below the effective threshold → exit 1; scan error → exit 2). `--out` is format-agnostic but introduced here for the SARIF/CI workflow.
  6. **Conformance validated against the official SARIF schema** — we do not author a Charter SARIF schema (SARIF is the external OASIS standard), but we vendor the official `sarif-2.1.0` JSON schema as renderer test data and validate every emitted document against it with a **test-only** dependency (`github.com/santhosh-tekuri/jsonschema/v6`, pinned at v6.0.2), alongside golden-file + structural tests. The validator is imported only from `_test.go` files, so it never enters the shipped binary. This is a deliberate, single exception to the dependency-minimalism posture, justified because SARIF is an external integration standard where schema conformance — not our own golden expectation — is the real contract.
- Consequences: new `internal/render/sarif` and `internal/rules/catalog` packages; `internal/version` gains a `Version()` function (2026 idiom: ldflags-injected value → else the toolchain-embedded `runtime/debug` module version, so `go install …@vX` is accurate → else `0.0.0-dev`); `cmd/charter/doctor.go` gains `--format sarif` and `--out`. The catalog + spec-sync test guard rule metadata against drift; the SARIF renderer reuses `findings.Location`, the ADR-0013 suppression source, and the `Informational` flag with zero model changes. A pinned **test-only** dependency (`github.com/santhosh-tekuri/jsonschema/v6`) plus a vendored `sarif-2.1.0` schema under the renderer's `testdata/` validate output conformance in CI; neither ships in the binary, and the repo's `:security` gate (osv-scanner/govulncheck) covers the dep. This unblocks the M1.5 GitHub Action (it just runs `charter doctor --format sarif --out charter.sarif` and uploads). Deferred: `charter explain` (which will reuse the catalog), the live `use-charter.dev/rules` docs pages behind the `helpUri`s, SARIF 2.2, and `artifacts[]`/`invocation` enrichment. Related: ADR-0001 (offline-first core), ADR-0004 (contract-first interfaces), ADR-0006 (latest-docs-first), ADR-0008 (score formula), ADR-0009 (finding location), ADR-0013 (suppression model).
