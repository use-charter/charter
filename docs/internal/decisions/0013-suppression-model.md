# ADR-0013 Suppression & Governance Model

- Status: Accepted
- Context: Charter's last three v1 rules are the governance rules `AE-SUPPRESS-001` (suppression missing `reason` — Medium), `AE-SUPPRESS-002` (permanent suppression missing `approver` — High), and `AE-SUPPRESS-003` (high suppression rate — informational, Medium-labeled, does **not** affect score). They require a suppression mechanism that does not exist yet. The canonical architecture (`charter-architecture-2026.md`) already commits to two suppression surfaces and an unfulfilled scoring contract: (1) §0.2 describes `AE-SUPPRESS-001` as a *"Suppression **comment** missing required `reason="…"`"* and `AE-SUPPRESS-002` as a *"Permanent suppression present without `approver="…"`"* — i.e. inline `key="value"` comment directives; (2) §1.8's command gallery shows `charter suppress AE-CC-001 --reason "…" --expires 90d` writing `.charter-suppress.yml`, with expired entries that *"re-surface as a finding on expiry"*; (3) the Score Formula and Hard Caps tables both state *"Suppressed findings are excluded from the base score; they are listed separately"* — but `scoring.Calculate` currently scores every finding. Three design tensions must be resolved: (a) inline comment scanning is false-positive-prone (a doc that merely *mentions* `charter:ignore` must not be treated as a directive); (b) suppression must not *silently* hide raw secrets — Commitment #9 forbids printing and destructive auto-fix of secrets, and the audit checklist treats secrets as suppressible only under governance (a reason, and an approver for a permanent waiver), so the control is governance + approver-gated permanence, not a non-suppressible rule class; (c) `charter suppress` writes a file, which brushes against the "no silent mutation / diff-first" constraint (ADR-0005).
- Decision:
  1. **Two suppression sources, one engine** — a new `internal/suppress` package loads suppressions from (a) `.charter-suppress.yml` at the repo root (external source; YAML via `gopkg.in/yaml.v3`, the existing dep) and (b) inline `charter:ignore` comment directives (in-source). The schema: `suppressions: [{rule, reason, expires?, approver?, path?}]`; `expires` absent ⇒ a default-TTL suppression that is honored, while only the literal `expires: permanent` is a permanent waiver (honoring governed by decision 3). The inline grammar is `charter:ignore <RULE-ID> reason="…"` with optional `expires=YYYY-MM-DD approver="…"`, recognized after a `#`, `//`, or inside an `<!-- … -->` line comment. Sources map to SARIF `suppression.kind`: inline ⇒ `inSource`, file ⇒ `external` (forward-compat for the later SARIF slice; not emitted in v1).
  2. **Inline directives are detected per-finding, not by repo-wide scan** — for each produced finding that carries a concrete `path:line`, the engine reads *that one line* of *that one file* and checks for a matching `charter:ignore <thatRuleID>` directive. This is O(findings), is deterministic, and structurally eliminates the documentation false-positive (a spec or README that mentions the syntax is never on a finding's own line). Inline suppression therefore applies only to line-anchored findings in comment-supporting files; file-level/absence findings (e.g. missing AGENTS.md) and strict-JSON configs (`.claude/settings.json`, `.mcp.json`) cannot be suppressed inline and must use `.charter-suppress.yml`. Same-line anchoring only; next-line/block forms are deferred.
  3. **Expiry, the 90-day default, and permanent approval** (aligned with the audit checklist's TTL model) — a suppression with an explicit `expires: YYYY-MM-DD` is time-bounded and re-surfaces as an active finding once past; an expired or malformed date is inert (**fail closed** — it suppresses nothing). A suppression with **no `expires`** is a default-TTL suppression and **is honored** — the audit's "90-day default," which the `charter suppress` CLI realizes as an explicit date at write time. Only the literal `expires: permanent` is a permanent waiver: it is **honored only when it carries an `approver`**, and a permanent waiver without an approver is **not honored** (the finding stays active) and is flagged by `AE-SUPPRESS-002` — matching the audit's "treated as not suppressed long-term." The engine compares dates against an injectable `now` (defaults to `time.Now`) for deterministic tests. The `charter suppress` CLI: `--expires` defaults to `90d`, accepts a duration or an absolute date (always **stored as absolute `YYYY-MM-DD`**), and `permanent` is accepted with a warning that it requires `--approver` and stays governance-flagged (full permanent enforcement is a Phase 2 Cloud concern). v1 does not synthesize a creation-anchored 90-day window for a hand-authored bare entry; a bare entry is honored as-is (the in-source/in-file review is the accountability), and the CLI's date default keeps generated entries time-bounded.
  4. **Secrets are suppressible under the same governance as any rule** — the canonical docs treat `AE-SEC-001/002` as suppressible (the audit checklist's `AE-SUPPRESS-002` fix example is `# charter:ignore AE-SEC-001 reason="…" expires=permanent approver="…"`), and Commitment #9 forbids *printing* raw secrets and *destructive auto-fix*, not suppression. A secret finding is excluded from the score (its ≤ 49 cap lifts) **only while its suppression is honored**; a time-bounded secret suppression re-fires on expiry and a permanent one is not honored without an `approver` (AE-SUPPRESS-002, High), so the ≤ 49 cap returns. There is **no non-suppressible rule class** in v1; the approver-gated permanent path is the control, not a hard-coded exception list.
  5. **Scoring excludes suppressed and informational findings** — the doctor pipeline partitions findings into **active** and **suppressed** *before* scoring, so `scoring.Calculate` only ever sees active findings (fulfilling the architecture's "excluded from the base score" contract, including the blocker/secret caps). A new `Informational bool` on `findings.Finding` marks score-neutral findings (`AE-SUPPRESS-003`); `Calculate` skips them when tallying severities. `doctor.Result` gains a `Suppressed` list so renderers can list suppressed findings separately.
  6. **Governance rules** — a new `internal/rules/governance` package emits: `AE-SUPPRESS-001` (Medium) for any audited suppression with a blank `reason`; `AE-SUPPRESS-002` (High) for any explicit permanent waiver (`expires: permanent`) with a blank `approver` (a bare entry with no `expires` is a default-TTL suppression and is not flagged); `AE-SUPPRESS-003` (informational) when the suppression rate `suppressed / (active-rule-findings + suppressed)` exceeds **0.30** (the audit checklist's 30% threshold; the denominator excludes governance findings to avoid a feedback loop). "Audited" suppressions are all non-expired `.charter-suppress.yml` entries plus inline directives discovered at a finding location (including permanent-unapproved ones, so 002 can flag them); expired/malformed-date entries are inert and not audited. Governance findings are evaluated *after* the partition and are themselves **not** suppressible in v1. The 003 threshold is a v1 constant; making it profile-tunable is deferred to the policy-profiles slice.
  7. **`charter suppress` is a Charter-owned-file writer, distinct from `charter fix`** — it only ever creates/updates `.charter-suppress.yml` (Charter-owned governance state), never user source files; it prints the exact entry written and the file path, and supports `--dry-run`. This is consistent with "no silent mutation" (the write is explicit and echoed) and is intentionally outside the diff-first `charter fix` flow (ADR-0005), which governs mutations to *user* files. An identical `rule`+`path` entry is replaced rather than duplicated. v1 writes rule-level entries (`--reason --expires --approver`); engine support for an optional per-finding `path:` scope exists for hand-authored entries, but a `--scope` flag on the command is deferred (keeps the command minimal and avoids overloading `--path`, which selects the repo as in `doctor`).
  8. **Canonical field names** — `reason`, `approver`, `expires` for both the YAML keys and the inline `key="value"` segments (per architecture §0.2 `approver="…"` and §1.8 `--expires`). The audit checklist's older variants (`ExpiresAt`, `ApprovedBy`, `approved-by=`) are reconciled to these as-built names during docs sync.
- Consequences: a new `internal/suppress` package (loader, inline directive parser, `Apply` partition with injectable clock, fuzz target on both parsers) and a new `internal/rules/governance` package (the three rules) land, wired into `internal/doctor/run.go` between rule evaluation and scoring; `internal/findings` gains `Informational`, `internal/scoring` skips informational findings, and `doctor.Result` gains `Suppressed`. All three renderers (`text`, `json`, `markdown`) gain a "suppressed (N)" section and surface informational findings without scoring them; the JSON payload gains a `suppressed` array and an `informational` field. A new `charter suppress` Cobra command lands in `cmd/charter`. The `AE-SUPPRESS-001/002/003` specs move from stub to as-built contracts, and the audit checklist rows (and HTML mirror) are reconciled to the as-built field names (`reason`/`approver`/`expires`) and the 30% rate threshold; with these three rules the **full 15-rule v1 catalog is complete**. Deferred for later slices: next-line/block inline forms, profile-tunable 003 thresholds (policy-profiles slice), `--format sarif` emission of `suppressions[]` (M1.5), and suppression of governance findings. Malformed `.charter-suppress.yml` fails the scan fast with a wrapped error, consistent with the offline-first, fail-fast posture. Related: ADR-0001 (offline-first core), ADR-0004 (contract-first interfaces), ADR-0005 (diff-first fixes), ADR-0006 (latest-docs-first), ADR-0008 (score formula), ADR-0009 (finding location).
