# ADR-0018 Charter GitHub Action & SARIF Upload

- Status: Accepted
- Context: The GitHub Action is Charter's **primary product surface** (architecture M1.5 T1.5.1) and the thing Phase 1 validation Signal 1 measures — strangers writing `uses: use-charter/charter-action@v1`. Slice 9 (ADR-0016) produced the rails the Action consumes: signed, multi-OS/arch binaries, a sha256 `checksums.txt`, a cosign keyless `checksums.txt.sigstore.json` bundle, and SLSA L3 provenance — but **armed, not fired** (no release tag cut yet). The `charter doctor` CLI already emits exactly what the Action needs: `--format sarif --out <file> [--threshold N]` with exit 0/pass, 1/below-threshold, 2/error (ADR-0014, Slice 9). `AE-CI-002` already credits `use-charter/charter-action@` as the Charter CI gate and flags non-SHA-pinned third-party actions. The repo carries an `action/` placeholder dir ("workspace contract for future GitHub Action packaging"). Verified live (2026-06-01): **`github/codeql-action/upload-sarif` is v4** (v4.35.1, Node 24; v3 deprecated Dec 2026) — the version GitHub Code Scanning advanced setup uses; **cosign v3 `verify-blob --bundle <file>.sigstore.json --certificate-identity <workflow-ref> --certificate-oidc-issuer https://token.actions.githubusercontent.com <file>`** is the keyless verification path (no `id-token` needed to *verify*, only to sign); GitHub **Marketplace actions must be a dedicated repo with `action.yml` at root** (a monorepo subdir cannot be listed and is referenced only via the clunky `owner/repo/path@ref`). T1.5.3 sets performance budgets: `charter doctor` ≤ 2 s wall-clock and ≤ 256 MB RSS on a 50,000-file repo, race-clean.
- Decision:
  1. **Composite action, developed in the monorepo `action/` dir as the single source of truth, seeded to a dedicated `use-charter/charter-action` repo at launch.** The consumer-facing ref stays the documented, branded `use-charter/charter-action@v1` (Marketplace-ready: `action.yml` at the dedicated repo's root). Developing in `action/` keeps it reviewed, tested, and version-controlled with the code it wraps and inside this slice's flow; the dedicated repo is populated by a **one-time manual seed** at launch (copy `action/` → tag `v1`). Automated monorepo→action-repo sync is deferred until the action churns enough to justify it. A subdir-only consumer path (`use-charter/charter/action@v1`) is rejected: off-brand, not Marketplace-listable, and would force an `AE-CI-002` rule change.
  2. **Action flow (composite steps):** resolve the requested charter version → map the runner OS/arch to the release asset → download the archive + `checksums.txt` + `checksums.txt.sigstore.json` → (when `verify`) install cosign via SHA-pinned `sigstore/cosign-installer`, `cosign verify-blob` the checksum bundle against the `release.yml` identity, then `sha256 -c` the archive against `checksums.txt` → extract → `charter doctor --path <path> --format <format> --out <sarif-file> [--threshold N]` capturing the exit code → (when `upload` and format is sarif) `github/codeql-action/upload-sarif@v4` → propagate the doctor exit code last (so a below-threshold gate fails **after** alerts upload).
  3. **Inputs (all optional, sensible defaults):** `version` (default `latest`), `path` (`.`), `threshold` (empty → defer to `charter.yaml` policy/default), `format` (`sarif`), `sarif-file` (`charter.sarif`), `upload` (`true`), `verify` (`true`), `fail-below` (`true`), `category` (empty). `verify: false` and `upload: false` are escape hatches (air-gapped / self-upload). **Outputs (as shipped):** `exit-code`, `score`, and `sarif-file`; `score` is derived from a fast follow-up JSON scan so downstream workflows can gate or annotate on the numeric result without parsing SARIF.
  4. **Full supply-chain verification is the default** (dogfooding Charter's own brand): cosign keyless `verify-blob` of `checksums.txt` with `--certificate-identity https://github.com/use-charter/charter/.github/workflows/release.yml@refs/tags/<version>` and `--certificate-oidc-issuer https://token.actions.githubusercontent.com`, then a sha256 check of the downloaded archive. Verification needs no consumer OIDC.
  5. **SARIF upload on by default;** the consumer grants `security-events: write` (+ `actions: read`/`contents: read` for private repos). Upload runs even on a below-threshold result so Code Scanning alerts always surface, then the action fails if `fail-below`.
  6. **Every third-party action is SHA-pinned** (`cosign-installer`, `upload-sarif@v4`, any `checkout`). Because zizmor/actionlint only scan `.github/workflows/` (not `action/action.yml`), a repo-owned validator (Bun TS, per the repo's helper-script convention) asserts the action's structure and that all its `uses:` refs are full-SHA-pinned; it runs in `:check`. (The action's runtime steps are portable **bash** — the "TS via Bun" rule governs repo dev helpers, not action steps that execute on arbitrary consumer runners.)
  7. **Armed, not fired** (mirrors Slice 9): the end-to-end download-a-real-release path is verified at the first signed release / Slice 19; this slice validates the action's structure + SHA-pins + documented usage, and lands the perf validation in full. Charter's own CI keeps the `go run ./cmd/charter doctor` gate (which already satisfies `AE-CI-002`) until the action repo is seeded at launch.
  8. **Perf validation (T1.5.3):** a test synthesizes a ~50,000-file repository in a temp dir at test time (not committed), runs the scan, and asserts ≤ 2 s wall-clock everywhere plus, on Linux (CI), peak RSS ≤ 256 MB read from `/proc/self/status` `VmHWM` (RSS is not portably readable in Go; the assertion is Linux-gated, with a `runtime.MemStats` proxy reported elsewhere). It is build-tagged / behind a dedicated Moon task so it does not slow the default `:test`.
- Consequences: `action/` gains `action.yml` (composite, branded), a `README.md` with a copy-paste usage block, the bash download/verify/run logic, and a Bun TS action validator wired through `action/moon.yml` into `:check`; a new perf fixture generator + perf test + Moon task land under the scanner. Docs update (`AGENTS.md`/`README.md`/`ARCHITECTURE.md`), the architecture T1.5.1 wording is reconciled to as-built (upload-sarif v4, full cosign+checksum verification), and the launch roadmap's Slice 17 checklist gains "create + seed `use-charter/charter-action` and switch Charter's own CI to dogfood it." No new dependency enters the Charter binary. Deferred: dedicated-repo creation/seed and first e2e (launch/Slice 19), automated sync, Marketplace listing, and dogfooding the action in Charter's CI. Related: ADR-0016 (release pipeline & identity), ADR-0014 (SARIF + catalog), ADR-0006 (latest-docs-first); architecture M1.5 T1.5.1/T1.5.3.
