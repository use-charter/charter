# ADR-0023 Agent-Operability Rules (Testing, Autonomy, Instruction Quality) & Category Scorecard

- Status: Accepted
- Context: Charter's v1 rule set (15 rules) scores two of the three things that make a repository agent-ready — **context** (does the agent have good orientation? AE-CTX/AE-CC) and **safety** (is the config safe? AE-SEC/AE-MCP/AE-CC) — but is blind to the third: **operability** (can the agent actually *verify its own work* and *run the project*?). This is the defining capability gap. Grounding (2026): the AGENTS.md standard (an open standard under the Agentic AI Foundation / Linux Foundation, adopted by 60k+ repos and 20+ agent tools) states that the **exact CLI commands an agent runs — install, test, lint, build — are "the single highest-value section"** of an agent-context file, and that agents **"attempt to execute relevant programmatic checks and fix failures before finishing a task."** A repo with no tests gives the agent nothing to verify against; a repo with no discoverable command contract gives it no way to run the checks. Separately, peer-reviewed instruction-following research (arXiv 2603.25015 *Imperative Interference*; the "instruction-complexity-cliff" analyses) shows that **stacked emphatic directives** (`IMPORTANT`/`NEVER`/`MUST`/`CRITICAL`) create a fragile, competitive instruction topology that *degrades* adherence, while concise, declarative guidance transfers more reliably — so over-emphasis is a measurable context-quality anti-pattern. Charter's AE-CTX-001 already requires that an agent context file *mention* a verification command and stay within budget; these rules verify the underlying reality. Score model: ADR-0008's deduction formula (`100 − B×20 − H×10 − M×4 − L×1`, informational excluded) is the stable public contract and is **unchanged** here.
- Decision: add three rules in two new categories plus one Context rule, and a reporting-only category scorecard. All detection is offline, deterministic, over the tracked inventory (no network, no LLM — Commitment #4/#7).
  1. **`AE-TEST-001` — Testing — High.** *Automated tests are present.* Stack-aware: for each language detected from a root manifest (`go.mod`→Go, `package.json`→JS/TS, `pyproject.toml`/`setup.py`/`requirements.txt`→Python, `Cargo.toml`→Rust, `pom.xml`/`build.gradle*`→Java/Kotlin, `Gemfile`→Ruby, `*.csproj`→C#, `composer.json`→PHP), require ≥1 recognized test artifact for that language (Go `*_test.go`; JS/TS `*.{test,spec}.*` / `__tests__/`; Python `test_*.py`/`*_test.py`/`tests/`/`conftest.py`; Rust `tests/**.rs` or a `#[test]`/`#[cfg(test)]` signal; Java/Kotlin `src/test/**`/`*Test.*`; Ruby `*_spec.rb`/`*_test.rb`/`spec/`/`test/`; C# `*Tests.cs`; PHP `tests/`/`*Test.php`). Fires when a detected language has zero test artifacts; **N/A** (no finding) when no recognized code language is present (docs/config-only repos). Severity **High** — an agent cannot self-verify changes without tests; not Blocker (the repo is still usable, just unverifiable). v1 judges *presence*, not coverage or pyramid shape.
  2. **`AE-AUTO-001` — Autonomy — Medium.** *A discoverable command contract for the dev loop exists.* The verification commands an agent runs are discoverable via **either** a standardized task runner — `Makefile` targets, `justfile` recipes, `Taskfile.yml` tasks, `package.json` scripts, `mise.toml [tasks]`, or `moon.yml` tasks — exposing a test entrypoint and a lint-or-build entrypoint, **or** the detected language's conventional zero-config toolchain (Go `go test`/`go build`; Rust `cargo test`/`cargo build`; etc.), which is inherently discoverable. **FP guard:** a single-language Go/Rust/Cargo repo is *not* penalized for lacking a `Makefile` — its toolchain is the contract. Fires only when neither a runner contract nor a conventional toolchain path makes the dev loop discoverable (e.g. a JS/Python repo with tests but no declared `test`/`lint` scripts, or a polyglot repo with no unifying runner). Severity **Medium**. Complements AE-CTX-001 ("a verification command is *mentioned*") by checking the command is *real and runnable*.
  3. **`AE-CTX-006` — Context — informational.** *Agent instructions are not over-emphasized.* Counts emphatic-directive density (`IMPORTANT`, `NEVER`, `MUST`, `CRITICAL`, `ALWAYS`, `EXTREMELY`, `ABSOLUTELY`, `FORBIDDEN`, `PROHIBITED` — word-boundary) per 1,000 words in the agent context file; flags when density exceeds a conservative threshold (≥15 / 1K words). **Informational** (`Informational: true` — re-surfaces in output, does **not** deduct, mirroring AE-SUPPRESS-003), so it is a quality nudge with zero scoring-FP risk. Depends on AE-CTX-001 (only when a context file exists). Grounded in the instruction-following research above.
  4. **Category scorecard (reporting only).** The report gains a per-category readiness breakdown — each category with its finding count, total deduction, and worst severity — so a score reads as a diagnostic scorecard ("Context −0 · Testing −10 · Autonomy −4 · …") rather than a bare number. The JSON adds a `categories` array; text/markdown gain a breakdown section. **The score formula (ADR-0008) is unchanged** — this is a presentation layer over the existing deductions, not per-dimension re-scoring (which would break the stable public score contract).
- Refinement (2026-06-02, FP-validation): scanning 10 real public repos surfaced two false positives, both fixed before sign-off (0% FP, recorded in `docs/internal/operability-fp-validation.md`). (a) "Active language" now requires a **project manifest** in addition to non-test source outside tooling dirs — a stray `pkg/brew/*.rb` Homebrew formula no longer activates Ruby in a Rust repo. (b) **Directory-based test detection** — a file under a `test/`/`tests/`/`spec/`/`__tests__/` segment counts as a test for any language — catches AVA/tap/node:test/RSpec layouts (e.g. tests in `test/*.ts`) that don't use a per-file `*.test.*` name.
- Refinement (2026-06-03, self-dogfood): embedding a browser asset (`internal/render/html/assets/report.js`) into the binary via `//go:embed` (Slice 16) surfaced a third FP — the single embedded `.js` plus Charter's build-tooling `package.json` wrongly activated JavaScript/TypeScript as an untested surface (Charter scored 90 instead of 100). (c) **Embedded-asset gate** — files referenced by a `//go:embed` directive in a tracked `.go` file are bundled resources of the host program, not independent language surfaces, and are excluded from the language source/test counts. Patterns resolve relative to the directive's `.go` file directory (exact files, directory subtrees recursively, and `path.Match` globs; a leading `all:` prefix is stripped). Because `//go:embed` is Go-only, the `.go` scan runs only when a non-Go manifest is present, preserving the 50k-file perf budget (Commitment #4/#7 unchanged: still offline, deterministic, stdlib-only). Charter is back to 100 with no AE-TEST-001 (recorded in `docs/internal/operability-fp-validation.md`).
- Consequences: two new rule packages (`internal/rules/testing`, `internal/rules/autonomy`) wired into `doctor.Run`; `AE-CTX-006` added to `internal/rules/context`; the rule catalog (`internal/rules/catalog`), the architecture §0.2 rule table (15→18 rules), and new `docs/internal/specs/AE-{TEST,AUTO}-001.md` + `AE-CTX-006.md` + audit-checklist entries. A category-breakdown renderer in `internal/render/*` + a `categories` field in the JSON DTO (additive; the SARIF golden and the score formula are untouched). Language/test detection reuses the manifest-reading pattern from `internal/scaffold`. Charter dogfoods to 100: it has Go tests (`AE-TEST-001` ✓), a `moon.yml` task contract (`AE-AUTO-001` ✓), and a declarative, non-shouty AGENTS.md (`AE-CTX-006` ✓). FP discipline (Commitment #9): `AE-TEST-001` is N/A for non-code repos and passes on any test file; `AE-AUTO-001` counts conventional toolchains so idiomatic Go/Rust repos aren't flagged; `AE-CTX-006` never deducts. Related: ADR-0008 (score formula, unchanged), the AE-CTX-001 "verification command" check these deepen, ADR-0013 (`Informational` precedent). This is the Slice 14 expansion that takes Charter from an agent-*setup* linter to an agent-*operability* scorecard.
