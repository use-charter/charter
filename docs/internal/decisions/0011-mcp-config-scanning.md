# ADR-0011 MCP Config Scanning

- Status: Accepted
- Context: Charter's v1 rule set includes three MCP-safety rules — `AE-MCP-001` (server packages must be pinned), `AE-MCP-002` (remote origins must be known or allowlisted), and `AE-MCP-003` (sensitive remote servers must declare auth metadata) — mapping to OWASP MCP Top 10 (beta) `MCP04`, `MCP09`, and `MCP07` and to the MCP specification revision `2025-11-25` (the latest stable revision; a `2026-07-28` release candidate is in flight and hardens OAuth but changes the wire protocol, not static config files). MCP server configuration exists across several client files and formats, the trusted-remote allowlist needs a home before the M1.6 catalog exists, and Charter's core has no YAML parser. These choices shape an interface (the `charter.yaml` config surface) and a dependency, so they are recorded here.
- Decision: (1) v1 scans **JSON** MCP config files only — `.mcp.json`, `mcp.json`, `.cursor/mcp.json`, `.vscode/mcp.json` — via the standard-library `encoding/json`; `mcp.yml` and Pkl MCP server scanning are deferred. (2) The `AE-MCP-002` trusted-remote allowlist lives in `charter.yaml` under `mcp.trustedRemotes` (a list of hostnames); when no allowlist is present, every non-local remote origin is flagged as unverifiable. (3) Charter adds `gopkg.in/yaml.v3` (latest, verified per ADR-0006) as its first non-Cobra runtime dependency, used only to read `charter.yaml`. (4) `AE-MCP-003` detects the **presence** of an authentication declaration (any auth header such as `Authorization`/`X-Api-Key`, even an env-reference) rather than asserting specific OAuth 2.1/PKCE field names, so it stays resilient to the MCP `2026-07-28` OAuth changes. (5) An unreadable or malformed MCP config or `charter.yaml` fails fast with a wrapped error, consistent with the offline-first, fail-fast posture; it is not silently skipped.
- Amendment (2026-06-14, Slice 19 launch-prep): the scanned-path set adds **`.gemini/settings.json`** (Gemini CLI's project MCP config; a mixed settings file from which only the `mcpServers`/`servers` object is read), and the remote-URL parser now also reads Gemini's **`httpUrl`** (streamable-HTTP transport) in addition to `url`. This closes a cross-vendor coverage gap surfaced by the MCP-catalog FP run 2 (a real `.gemini/settings.json` with unpinned packages went undetected) and aligns scanning with the README's Gemini support claim. Verified against the official Gemini CLI docs (`mcpServers` with `command`/`args`/`env` stdio, `url` SSE, `httpUrl` streamable HTTP). The original decision's intent is unchanged — JSON MCP configs via stdlib `encoding/json`, fail-fast on malformed input; this only extends the recognized filename + transport-field set. `mcp.yml` / Pkl remain deferred.
- Consequences: two new packages enter the tree — `internal/rules/mcp` (discovery, parsing, the three checks) and `internal/config` (the `charter.yaml` loader); `charter.yaml` becomes a recognized, optional config surface that future profiles can extend; `gopkg.in/yaml.v3` enters `go.mod`; the three `AE-MCP-*` specs move from stub to as-built behavioral contracts; MCP findings emit structured `path:line` locations per ADR-0009 and are `High` severity with no score cap; the M1.6 MCP catalog will later supersede the local `charter.yaml` allowlist as the primary origin-trust source; and a follow-up re-validates `AE-MCP-003` against the MCP `2026-07-28` revision once it ratifies. Localhost remotes are exempt from `AE-MCP-002`/`AE-MCP-003`. Related: ADR-0001 (offline-first core), ADR-0004 (contract-first interfaces), ADR-0006 (latest-docs-first), ADR-0009 (finding location).
