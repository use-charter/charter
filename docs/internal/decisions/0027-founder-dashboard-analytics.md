# ADR-0027 Founder-Dashboard Analytics — First-Party, Server-Side, D1-Backed

- Status: Proposed
- Context: The founder mission-control dashboard (shipped Slice-19-era, served by `charter-router` at `/dashboard/api/stats`, gated by Cloudflare Access — ADR-0026) currently surfaces **GitHub** signals only (stars, traffic, releases, adoption, community). It shows nothing about the **website** itself. The founder wants seven traffic signals on the same dashboard: visits/day, unique visits/day, per-page view counts, blog views, install-command copies (the primary conversion), docs click-through, and a top-5 visitor-country list. The tension: Charter's public brand is **offline-first, zero-telemetry** — but that posture (ADR-0001) governs the **CLI/core**, not the marketing site; first-party site analytics is a separate concern that must be delivered **without** betraying that ethos (no third-party tracker, no cookies, no PII, no consent-banner surface). The existing edge topology is the key enabler: per ADR-0026 the static Astro site lives on **Cloudflare Pages** behind the **`charter-router`** Worker, which owns a `use-charter.dev/*` catch-all and already proxies *every* request — `/docs`,`/rules` → Mintlify, everything else → the Pages origin — and already serves the Access-gated dashboard API. The router is the **only** component that already observes all traffic, and it is already serverless. Verified live against official sources (2026-06-19, ADR-0006): **Cloudflare Web Analytics** is a RUM pageload beacon — it supports **no custom events**, reports **"visits" (not device-unique visitors)**, and is **adaptive-sampled** ([web-analytics](https://developers.cloudflare.com/web-analytics/)); the incoming-request **`cf` object exposes `cf.country`** (ISO-2) to every Worker at no extra cost ([request.cf](https://developers.cloudflare.com/workers/runtime-apis/request/#incomingrequestcfproperties)); **D1 Free** = 5M rows read/day, **100k rows written/day**, 5 GB total ([D1 pricing](https://developers.cloudflare.com/d1/platform/pricing/)); **Workers Free** = 100k requests/day (reset midnight UTC), 10 ms CPU/invocation, 50 subrequests/request, `ctx.waitUntil()` up to 30 s after response ([Workers limits](https://developers.cloudflare.com/workers/platform/limits/)); **Workers Analytics Engine** has no stated Free-plan allowance in current pricing; **Workers Observability/Logs** Free = 200k events/day, 3-day retention ([Workers Logs](https://developers.cloudflare.com/workers/observability/logs/workers-logs/)).
- Decision: Add **first-party, server-side analytics** to the existing `charter-router` Worker, backed by a single **D1** database. No third party, no client-side rendering change, no Pages→Workers migration.
  1. **First-party server-side counting, not a third-party tracker and not CF Web Analytics alone.** The router counts each qualifying request as it already passes through it. We **reject** Google Analytics / Plausible / PostHog (a third-party dependency, an injected script, and a privacy/ethos conflict for a project that sells "no telemetry"). We **reject CF Web Analytics as the source of record** because it cannot capture custom events (install-copy) and its "visits" are not device-uniques; it may be layered in **later** only as an optional bot-filtered Core-Web-Vitals cross-check, never as the dashboard's data source.
  2. **One D1 database (`charter_analytics`) plus one KV namespace, bound to `charter-router`.** Four aggregate tables keyed by UTC day — `pageview(day,path,hits)`, `visitor(day,vhash)`, `geo(day,country,hits)`, `event(day,type,vhash)` — declared `WITHOUT ROWID` (the primary key *is* the storage, halving rows-written by removing the rowid index) and written via upsert / insert-or-ignore so every metric is a `GROUP BY`/`COUNT`. We **reject Analytics Engine for counting** (no stated Free-plan allowance — Charter must stay on free tier). The KV namespace holds **only the rotating daily salt** (1 write/day, ~hundreds of reads/day — within KV free limits), not per-request counters. D1's 100k-writes/day budget, after the `WITHOUT ROWID` reduction (~2 writes/upsert × 3/pageview), leaves **≈16k pageviews/day** headroom — ample for launch; the degradation path is aggregate-only (drop the visitor table) then write sampling.
  3. **Privacy model — cookieless, no stored PII, anonymized after 24h.** Visitor uniqueness is a truncated `SHA-256(dailySalt ‖ "use-charter.dev" ‖ client-IP ‖ user-agent)`, where `dailySalt` is a **random 32-byte value generated once per UTC day, stored in KV, and deleted within ~48h**. The raw IP and user-agent are used transiently in memory for the hash only and are **never** written to D1, KV, logs, or disk; no query strings or per-visitor path trails are stored either. We **reject a persistent/derived secret** (e.g. HMAC keyed by a long-lived secret + date): per EDPB pseudonymisation guidance a keyed hash with a durable key remains personal data — only **salt deletion** severs the link and supports treating the daily data as anonymous (the Plausible/Fathom model). Legal basis is **legitimate interest (GDPR Art. 6(1)(f))** with a published privacy-policy disclosure; no cookie/local-storage means the ePrivacy Art. 5(3) consent trigger does not apply. (This "no consent" posture is industry-standard but legally contested — the disclosure is mandatory.) Site-only and founder-only (read API Access-gated); the CLI/scanner core stays **zero-telemetry** (ADR-0001, unchanged).
  4. **Exactly one client beacon.** The install-command copy is a clipboard action with no navigation, so it cannot be observed server-side; it posts to a type-allowlisted `POST /api/event` on the router. **Every other metric is derived from server-side request observation** (`URL.pathname`, `request.cf.country`) with zero client JS. Because `/docs` is router-proxied, docs reach is measured as server-side `/docs` hits — no outbound-link beacon needed.
  5. **The site stays static; the dynamic layer stays in the router.** Reaffirms ADR-0026: no migration to Workers Static Assets, no SSR added to the Astro build. The only new resource is the D1 database; the router was already a serverless Worker fronting all traffic.
  6. **Reads via a new Access-gated `GET /dashboard/api/analytics`** that mirrors the existing stats handler's defense-in-depth (Access assertion header), returning the pre-aggregated series the dashboard renders. It **does not use the Cache API** — that API is unavailable for Access-fronted Workers (which also means the existing `/dashboard/api/stats` `caches.default` calls are effectively no-ops, noted for later cleanup) — so freshness comes from a short module-level in-isolate memo over direct D1 reads.
- Consequences: Two new Cloudflare resources — a D1 database (`[[d1_databases]]`) and a KV namespace (`[[kv_namespaces]]`, daily-salt only) declared in `infra/router/wrangler.toml` with IDs dashboard-set (extending the IaC reproducibility posture of ADR-0026 §4); the D1 schema lives as a checked-in Wrangler migration. The build stays within Free tier: per qualifying request it issues the three D1 writes as one `ctx.waitUntil()` `batch()` (no added response latency), one short SHA-256 hash, an `isbot` user-agent check, and at most one memoized KV salt read — all sub-millisecond, inside the 10 ms CPU/invocation limit; the binding ceiling that matters is D1's 100k writes/day (≈16k pageviews/day after `WITHOUT ROWID`), and the request ceiling (100k/day) is already the whole-site limit on Free, unchanged by this work. A degradation path exists if traffic outgrows free limits: drop the per-visitor table (aggregate-only) or head-sample writes. Honesty constraints are carried into the UI labels: "visits" = server-side pageviews/day (not sessionized), "uniques" = daily distinct hashed visitor (an approximation), and server-side counts include bot traffic beyond a coarse user-agent filter (CF Web Analytics auto-filters bots — the rationale for optionally layering it later). The trust posture shifts slightly — Charter now stores visitor-*derived* data for the first time — but the no-PII, daily-salted, founder-only design keeps the public privacy stance defensible, and the offline-first core (ADR-0001) is untouched. Out of scope / deferred: sessionized "visits", a data-retention/pruning policy (raw daily rows are negligible in 5 GB), the optional CF Web Analytics CWV overlay, and the **separate** Workers-Observability move (folding `web/functions/api/waitlist.ts` into the router with `[observability] enabled = true`) — recorded as a related future change, not part of this ADR. Related: ADR-0001 (offline-first, zero-telemetry core — explicitly preserved), ADR-0026 (edge topology + `charter-router` this extends; static-Pages decision reaffirmed), ADR-0006 (latest-docs-first — all Cloudflare limits/behaviors verified 2026-06-19), ADR-0024 (terminal/dashboard design system the new tiles inherit).
