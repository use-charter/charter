# ADR-0015 Policy Profiles

- Status: Accepted
- Context: The architecture (T1.1.3, T1.1.5) requires that teams gate pipelines on an objective threshold configured in `charter.yaml` rather than per-invocation flags: "a `charter.yaml` in the repo root with `policy: profile: strict` … `charter doctor` runs without any flags … Charter uses the strict profile; no flags required." Today `charter doctor`'s `--threshold` is a Cobra flag that defaults to `80`, and `internal/config` only loads `mcp.trustedRemotes` — there is no profile concept and no way to set the gate from config. `charter init` (M1.4) is also specified to write `charter.yaml` with a `policy` profile, so the config contract needs to exist first.
- Decision:
  1. **`charter.yaml` gains a `policy` block** — `policy: {profile: strict|standard|relaxed, threshold: <int 0..100>}`. Both keys are optional. The config loader (`internal/config`) is extended to parse it (reusing `gopkg.in/yaml.v3`).
  2. **Built-in profiles map a name to a threshold**, aligned to the architecture's score zones (80 = ship-ready boundary): `strict = 90`, `standard = 80`, `relaxed = 60`. These are fixed v1 constants.
  3. **Effective-threshold precedence** (highest wins): an explicit `--threshold` flag (detected via Cobra's `Changed("threshold")`, so a defaulted flag does not count) > `policy.threshold` (explicit numeric override) > `policy.profile` (named) > the `standard` default (80). An unknown profile name or an out-of-range `policy.threshold` **fails fast** with a wrapped error (exit 2), consistent with the offline-first, no-silent-failure posture.
  4. **v1 is threshold-only.** Profiles map solely to the passing threshold. Rule enable/disable, per-rule severity overrides, and `rules.ignore` are a larger policy engine and are deferred to a later slice — they are not part of the profile contract now.
  5. **Resolution lives in the scan pipeline, not the flag layer.** `doctor.Run` (which already resolves the repo root and builds the inventory) loads the policy and resolves the effective threshold, so the resolved value flows into `Result.Threshold`, `Passed`, every renderer, and the exit code uniformly. `Run`'s signature becomes `Run(path string, threshold int, thresholdSet bool)`; the CLI passes the flag value and `cmd.Flags().Changed("threshold")`. The resolver is a small pure function (`internal/policy` or a `config` helper): `Resolve(cfg, flagThreshold, flagSet) (int, error)`.
  6. **A contract-first config schema lands** — `schemas/charter-config.schema.json` (`$id: https://use-charter.dev/schemas/charter-config.schema.json`) documents `mcp.trustedRemotes` and `policy.{profile,threshold}`, matching the existing `doctor-result.schema.json` convention (ADR-0004). The resolved profile/threshold may also be stamped into the SARIF `run.properties` for traceability (optional, non-normative).
- Consequences: `internal/config` gains `policy` loading and a built-in-profile table; a small `Resolve` function owns precedence; `doctor.Run` takes a `thresholdSet bool` (callers — `cmd/charter` and the doctor tests — updated accordingly); `Result.Threshold` becomes the *effective* threshold; new `testdata/repos/` fixtures carry a `charter.yaml` `policy` block; `schemas/charter-config.schema.json` is added. Behavior is backward-compatible: with no `charter.yaml` policy and no flag, the gate stays 80. No new `AE-*` rule (profiles are configuration, not a finding). Deferred: rule-level enable/disable + severity overrides; XDG/global `charter.yaml` precedence (T1.1.5 — v1 reads the repo-root file only); a `CHARTER_THRESHOLD` env override; and `charter init` writing the `policy` block (M1.4). Related: ADR-0004 (contract-first interfaces), ADR-0006 (latest-docs-first), ADR-0008 (score formula), ADR-0014 (SARIF output — shares the CLI surface and may carry the resolved profile in `run.properties`).
