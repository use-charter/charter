# ADR-0021 MCP Catalog v1 — Severity Model, Allowlist & Curation

- Status: Accepted
- Context: M1.6 (the "Recurring Engagement Loop") specifies a *static, founder-curated MCP catalog* so `AE-MCP-001`/`AE-MCP-002` reference a versioned list of known servers and **re-fire on repos that previously passed** when a dependency receives a CVE or a new stable release — the architecture's named remedy for the "one-time setup trap" failure mode ("Some installs, zero stranger issues → Accelerate the static MCP catalog"). T1.6.1's worked example assumes semver (`@modelcontextprotocol/server-filesystem@1.0.2` → stable `1.0.4`) and fires **HIGH** for version-lag. Grounding against the live MCP ecosystem (2026-06) invalidated both assumptions:
  1. **The official servers use CalVer, not semver.** `@modelcontextprotocol/server-filesystem` is at `2026.1.14` (after `2025.12.18`, `2025.11.25`); it jumped from `0.6.x` semver straight to date-based versioning. A naive "is X < Y" comparator is fragile across semver/CalVer/date schemes and would itself be a false-positive source.
  2. **The catalog's most valuable signal is not version-lag — it is deprecation.** The official reference set shrank to **7 servers** (filesystem, fetch, git, memory, sequentialthinking, time, everything); the rest — `server-github`, `server-postgres`, `server-gitlab`, `server-slack`, `server-gdrive`, `server-sqlite`, `server-puppeteer`, `server-brave-search`, `server-redis`, `server-aws-kb-retrieval` — were **archived** and migrated to first-party vendor builds (`github/github-mcp-server` at `api.githubcopilot.com`) or community forks. Pinning to an archived package is a genuine, *staleness-proof* maintenance/security signal (archival is a fact that does not regress, unlike a fast-moving version number).
  3. **Vendor remote servers converged on a small set of hosted OAuth 2.1 endpoints** (`api.githubcopilot.com`, `mcp.sentry.dev`, `mcp.linear.app`, `mcp.atlassian.com`, `mcp.notion.com`, `mcp.stripe.com`, `mcp.vercel.com`, `mcp.asana.com`), which a baseline allowlist can recognize to reduce `AE-MCP-002` false positives.

  Charter's defining promise is Commitment #9 (false-positive rate below a measured threshold, enforced by zero-FP clean-repo fixtures). A founder-curated catalog tracking a 13k-server ecosystem **will** lag reality; the design must make staleness *safe* (a stale catalog stays silent, never wrong) rather than load-bearing.

- Decision:
  1. **Split severity by signal class — version-lag is informational, security is HIGH.** This reconciles the architecture's T1.6.1 "HIGH for version-lag" to the FP-first reality:
     - **Deprecated/archived package** → `AE-MCP-001` **HIGH** (deducts): *"`<pkg>` is archived; migrate to `<successor>`."* Stable, high-value, never goes stale.
     - **Known advisory (CVE/GHSA) affecting the pinned version** → `AE-MCP-001` **HIGH** (deducts): *"`<pkg>@<v>` is affected by `<id>`; fixed in `<fixed_in>`."* A real, durable security signal — the genuine "act now" engagement moment.
     - **Behind the catalog's `stable_version`, no advisory** → `AE-MCP-001` **informational** (`Informational: true`, re-surfaces, **zero** score impact), mirroring the existing `AE-SUPPRESS-003` idiom. The recurring-engagement value comes from *visibility* on the next scan, not from punishing a repo that did the right thing (an exact pin one patch behind). This matches Dependabot/Renovate convention (freshness = low-stakes nudge; CVE = alert) and bounds the staleness blast radius: a lagging catalog produces at worst an annoying note, never a failed CI gate.
  2. **Exact-match comparison only — no cross-scheme version ordering.** Each active server entry carries an ascending `known_versions` list whose last element equals `stable_version`. The behind-stable nudge fires **only** when the pinned version is *in* `known_versions` and is not the last and is not covered by an advisory. A version absent from `known_versions` (newer than the catalog, a private fork, an unrecognized scheme) is **silent** — the safe direction. Advisory matching is exact-version membership in `affected[]`. No `<`/`>` across version strings anywhere; CalVer, semver, and date schemes are all handled by string equality against curated sets.
  3. **One finding per server, precedence-ordered.** Per `(file, server)` `AE-MCP-001` emits at most one finding: **deprecated** > **unpinned** (the existing pin-mechanics HIGH) > **advisory** > **behind-stable** > clean. Deprecation outranks "not pinned" because pinning an archived package is moot — migration is the real fix.
  4. **`AE-MCP-002` gains the catalog `trustedHosts` as a baseline allowlist.** The effective allowlist is `union(charter.yaml mcp.trustedRemotes, catalog.trustedHosts)`. Recognized vendor-hosted origins now pass by default (pure FP reduction); unknown remotes still flag. `charter.yaml` remains the per-repo override. This keeps `AE-MCP-002`'s message/severity unchanged; only the baseline set widens.
  5. **The catalog is embedded, validated-at-build, never a runtime dependency.** Curated YAML at `internal/catalog/data/catalog.yaml`, embedded via `//go:embed`, parsed once and exposed read-only via `catalog.Default()`. A table-driven **catalog-validity test** is the curation contract: it asserts well-formedness (unique packages, lowercase hosts, `stable_version` == last `known_versions` for active entries, each advisory's `affected`/`fixed_in` non-empty, deprecated entries name a `successor` and carry no `stable_version`). A malformed catalog is a **test/build failure**, never a `charter doctor` runtime error — consistent with the offline-first, fail-fast, no-network, no-LLM core.
  6. **Curation is a founder responsibility with a documented process and a pre-ship FP gate.** T1.6.2 (contribution + CVE-advisory process, 48h founder SLA in Phase 1; community PRs + automated monitoring in Phase 2) and T1.6.3 (the ≤10% FP validation against 5+ real public repos, recorded in the FP-validation doc) are `⚑ FOUNDER` tasks. The seed ships with accurate package names and successor/host data (which age well) and best-effort version data flagged for a founder refresh + FP-validation pass before the public tag (Slice 17).
  7. **Deferred (unchanged from CF-6):** the `AE-MCP-001` *fixer* (auto-pin `@latest`/range → catalog `stable_version`, or rewrite a deprecated package to its successor) stays out of `charter fix` v1 — the catalog now supplies the pin/migration target, but content-aware MCP rewrites are a `fix` v1.1 item. No new runtime dependency; no MCP-003 change.

- Refinement (2026-06-02, from the T1.6.3 FP-validation run): two changes landed while validating against 11 real public repos (0% FP after the fix; recorded in `docs/internal/catalog/fp-validation.md`). (a) **Advisories may use `affectedBelow`** — the natural "fixed in X, affected below X" CVE shape — matched by a conservative component-wise numeric comparator (`versionLess`, correct for CalVer *and* semver; any non-numeric/unparseable version never matches). This bounded ordering applies **only** to authoritative CVE advisories, where range data is stable and high-value; the behind-stable nudge stays strictly exact-match. Real `mcp-server-git` CVEs (CVE-2026-27735 / CVE-2025-68145 / CVE-2025-68143, CWE-22) were seeded accordingly, fulfilling the M1.6 "CVE re-fires" user story with real data. (b) **`AE-MCP-002`/`AE-MCP-003` now exempt private/internal origins** (`isLocalHost`): RFC1918, link-local, the unspecified address, and the reserved `.localhost`/`.local`/`.internal` TLDs join loopback as non-public — a LAN/internal server is not a public shadow origin (fixed the single FP). Both stay within the offline/exact-or-bounded-numeric, no-network posture.
- Consequences: new `internal/catalog` package (embedded data + `Default()`/`Parse()` + `Lookup`/`AdvisoryFor`/`KnownBehind`/`TrustedHostSet` + validity test); `internal/rules/mcp/mcp001.go` becomes catalog-aware (`checkPinning(files, cat)` with the precedence ladder) and `mcp.go` merges `catalog.TrustedHostSet()` into the `AE-MCP-002` allowlist; new testdata fixtures (deprecated-package repo, behind-stable repo, catalog-trusted-host repo) and unit tests for each facet built against an inline test catalog (not `Default()`, to stay curation-stable). The rule-catalog metadata (`internal/rules/catalog`) broadens the `AE-MCP-001`/`AE-MCP-002` descriptions to mention catalog currency. Clean-repo fixtures stay zero-FP: the existing `1.0.4` pins are absent from `known_versions`, so they remain silent. The architecture's M1.6/T1.6.1 worked example and the `AE-MCP-001`/`AE-MCP-002` rule-table rows are reconciled to the CalVer reality and the informational/HIGH split, with this ADR carrying the rationale. Docs: `AE-MCP-001`/`AE-MCP-002` specs + the audit checklist updated; a catalog-contribution process doc (T1.6.2) and an FP-validation skeleton (T1.6.3) added; carry-forward CF-6 annotated (catalog available, fixer still deferred) and a new item tracks the founder refresh/FP-validation launch gate. Related: ADR-0011 (MCP config scanning — the `mcp.trustedRemotes` allowlist this extends), ADR-0008 (score formula — catalog HIGHs deduct like any HIGH; informational findings do not), ADR-0013 (`Informational` precedent via `AE-SUPPRESS-003`), ADR-0005/0020 (`fix` deferral); architecture §M1.6 T1.6.1–T1.6.3. The informational/exact-match design continues the dogfooding-driven, FP-first line (Commitment #9) from Slices 9/11/12.
