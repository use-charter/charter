**Rule ID:** AE-MCP-003 · **Severity:** High · **Category:** MCP Safety · **Auto-fixable:** No

## Why this rule

A public remote MCP server with no declared auth boundary accepts tool calls from any agent that can reach it. Charter checks whether the config declares that auth is required — the minimum signal that the server was configured with access control in mind.

## What triggers it

Charter scans tracked MCP configuration files — `.mcp.json`, `mcp.json`, `.cursor/mcp.json`, and `.vscode/mcp.json` — and inspects every remote HTTP or SSE server entry. For each non-local remote server, Charter checks whether an auth header is declared in the server's config. Accepted header names are `Authorization`, `X-Api-Key`, `Api-Key`, and `X-Auth-Token` (checked case-insensitively). An environment variable reference such as `"Bearer ${TOKEN}"` counts as a valid declaration — Charter checks for the presence of an auth header, not the validity of the credential.

A remote server with none of these headers declared is flagged High.

Two categories of server are always exempt: local and internal origins (loopback, RFC 1918 private addresses, `.localhost`, `.local`, `.internal`), and catalog-known OAuth vendor hosts such as Sentry, Atlassian, and Context7. These vendors authenticate through an OAuth flow rather than a static config header, so requiring a header declaration on them would be a false positive.

## Examples

### Failing

A `.mcp.json` entry for `"url": "https://self-hosted.example/mcp"` with no `headers` block — Charter cannot confirm auth is configured and flags it High.

### Passing

A server entry for `"url": "https://mcp.sentry.dev/mcp"` with no headers — Sentry is a catalog-known OAuth host and is exempt. Any remote server with `"headers": { "Authorization": "Bearer ${TOKEN}" }` also passes.

## How to fix

Add a `headers` block to the remote MCP server entry and include an auth header that references an environment variable:

```json
"headers": {
  "Authorization": "Bearer ${MCP_SERVER_TOKEN}"
}
```

If the server uses a different auth mechanism, declare the appropriate header (`X-Api-Key`, `Api-Key`, or `X-Auth-Token`) using the same env-reference pattern. Commit the change so the auth declaration is tracked.

## Score impact

each finding is `High` (−10); no hard cap.

## Edge cases

local/internal origins never fire (loopback, RFC1918 private, link-local, `.localhost`/`.local`/`.internal`); catalog OAuth vendor hosts never fire; an env-reference header value satisfies the presence check (Charter does not validate the credential, only its declaration); a bare `${VAR}` URL with no parseable host is skipped.
