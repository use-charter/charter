**Rule ID:** AE-CC-001 · **Severity:** Blocker · **Category:** Agent Config · **Auto-fixable:** No

## Why this rule

Agent hook configurations run shell commands automatically on events like file save or tool use. A destructive or injection-vulnerable hook fires without explicit user action, and in an agent session the agent may trigger it indirectly through file writes.

## What triggers it

Charter scans agent hook configuration files — `.claude/settings.json` and `.claude/settings.local.json` for Claude Code, and `.cursor/hooks.json` for Cursor — and inspects every command declared across all hook events.

It flags commands that fall into three danger classes:

- **Shell injection vectors:** command substitution via `$(…)` or backtick syntax, and `&&` or `||` chains where the left-hand side accepts open input that an injected prompt could control.
- **Destructive commands:** `rm -rf`, `git reset --hard`, `git clean -fd`, `dd`, `mkfs`, and `truncate` — operations that irreversibly delete or overwrite data.
- **Privilege escalation:** `sudo`, `chmod 777`, and `chown -R` — commands that widen permissions beyond what the task requires.

A hook running any of these patterns fires the finding at Blocker severity. Charter quotes the offending command in the finding evidence so you can locate it immediately.

## Examples

### Failing

`.claude/settings.json` with a hook `command` of `"rm -rf $CLAUDE_PROJECT_DIR/build"` — destructive, flagged Blocker. A hook using `"sudo chmod 777 ./bin"` is also flagged.

### Passing

`.claude/settings.json` with a `PreToolUse` hook running a scoped shell script — for example `"$CLAUDE_PROJECT_DIR/.claude/hooks/format.sh"` — passes because it delegates to an explicit, bounded command.

## How to fix

replace the dangerous command with an explicit, scoped, non-destructive command; prefer array-form (`args`) execution to avoid shell expansion; review each hook against "if an agent were prompt-injected, could this hook be weaponized?", then commit the change.

## Score impact

`Blocker` — engages the hard blocker score cap (final score ≤ 59 whenever any blocker finding is present); no separate per-finding cap.

## Edge cases

only the three JSON hook files are scanned in v1; Pkl (`hk.pkl`), YAML (`.pre-commit-config.yaml`, `lefthook.yml`), and shell-dir (`.husky/`) hook managers are out of scope for v1; a controlled `&&` chain (`cd app && npm test`) is not flagged (operator chaining is deferred); a malformed JSON hook file fails the scan fast with a wrapped error.
