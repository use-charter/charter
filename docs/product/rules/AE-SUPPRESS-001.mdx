title: "AE-SUPPRESS-001"
description: "Every suppression must carry a human-readable `reason` so that an accepted finding is auditable and a future reviewer understands why it was muted."
---

**Rule ID:** AE-SUPPRESS-001 · **Severity:** Medium · **Category:** Governance · **Auto-fixable:** No

## Why this rule

A suppression without a reason is indistinguishable from accident. When a finding re-surfaces after a security review or team rotation, there is no record of why it was acceptable or who decided.

## What triggers it

Charter audits every suppression that was applied during the current scan — all entries in `.charter-suppress.yml` and every inline `# charter:ignore` comment that matched a finding. For each applied suppression, it checks whether a `reason` field is present and non-empty. A YAML entry with no `reason:` key and an inline directive with no `reason="…"` segment both fire the finding.

The suppression itself still takes effect — Charter continues to suppress the underlying finding. A missing reason is a governance hygiene problem, not grounds to un-suppress.

## Examples

### Failing

A `.charter-suppress.yml` entry of `{rule: AE-CC-001, expires: 2026-08-16}` with no `reason` field — flagged Medium. An inline `# charter:ignore AE-MCP-001` comment with no reason — also flagged. Charter names the suppressed rule and its source file in the finding evidence.

### Passing

```yaml
- rule: AE-CC-001
  reason: "Claude config lives in the infra repo"
  expires: 2026-08-16
  approver: alice
```

An inline `# charter:ignore AE-MCP-001 reason="vendored test fixture"` also passes.

## How to fix

Add a meaningful `reason` to the suppression entry or inline directive that explains why the finding is accepted. For YAML entries, add a `reason:` field. For inline comments, add `reason="…"` after the rule ID. The `charter suppress <RULE> --reason "…"` command always writes the reason for you.

## Score impact

`Medium` — deducts 4 points via the standard formula; no cap.

## Edge cases

a suppression entry whose rule never matched any finding this scan and that came from `.charter-suppress.yml` is still audited (non-expired file entries are always evaluated; expired entries are inert and not audited); a dangling inline directive that matched no finding is **not** audited (inline directives are discovered only at finding locations); a `reason` of only whitespace counts as missing; secrets are suppressible like any rule, so a reason-less `AE-SEC-001/002` suppression is audited the same way.
