description: "A repository's active code language(s) must have automated tests. **Rule ID:** AE-TEST-001 · **Severity:** High · **Category:** Testing · **Auto-fixable:** No

## Why this rule

An agent that writes code in a language with no automated tests has no way to verify its own work. Tests are the agent's exit condition — without them it can only produce a diff, not a proof.

## What triggers it

Charter detects active languages from manifest files committed to the repo — `go.mod` for Go, `package.json` for JavaScript/TypeScript, `Cargo.toml` for Rust, `pyproject.toml` or `requirements.txt` for Python, `pom.xml` or `build.gradle` for Java/Kotlin, and equivalents for Ruby, C#, and PHP.

A language is only considered active when its manifest is present **and** the repo contains at least one non-test source file for that language outside tooling directories (such as `scripts/`, `vendor/`, `node_modules/`, `dist/`, and `build/`). This prevents a `package.json` that only drives build scripts in a Go repo from triggering a JavaScript test requirement, and it prevents embedded web assets in a Go binary from activating a JavaScript surface.

For each active language, Charter then checks whether at least one test file exists. It recognizes standard test conventions: `*_test.go` for Go, `*.test.*` and `*.spec.*` for JavaScript/TypeScript, `test_*.py` / `*_test.py` / `conftest.py` for Python, files under `tests/` for Rust (as well as inline `#[cfg(test)]` markers), `*Test.java` / `*Test.kt` for Java/Kotlin, and equivalents for other supported languages. Files in directories named `test/`, `tests/`, `spec/`, or `__tests__/` also count, regardless of language.

## Examples

### Failing

A Go module with source files under `internal/` but no `*_test.go` file anywhere in the repo — flagged High with evidence: `no test files detected for active language: Go`.

A TypeScript project with `src/index.ts` and a `package.json` but no `*.test.ts` or `*.spec.ts` files and no `__tests__/` directory — flagged High.

### Passing

A Go module with `go.mod` and at least one `*_test.go` file — passes. A TypeScript project with a `package.json` and files matching `*.test.ts` — passes. A Rust crate where source files contain `#[cfg(test)]` blocks — passes.

## How to fix

add tests for the active language(s) so an agent can run them and self-verify before finishing a task.

## Score impact

`High` (−10); no hard cap.

## Edge cases

**N/A** (no finding) when no recognized code language is active (docs/config/tooling-only repos). Tooling-only manifests (a `package.json` with only `scripts/*.ts`) and stray secondary-language files (a lone `*.rb` Homebrew formula with no `Gemfile`) are not active. A `//go:embed`'d web asset (e.g. a `report.js` embedded into a Go binary) is a bundled resource, not a language surface, so it does not activate JavaScript/TypeScript even when a build-tooling `package.json` is present. Rust inline unit tests count via the `#[cfg(test)]`/`#[test]` content signal. v1 judges *presence*, not coverage, quantity, or pyramid shape. Validated at 0% FP across 10 real public repos plus Charter's own self-dogfood.
