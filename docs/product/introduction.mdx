Teams don't fail at AI-assisted development because the model is bad. They fail because the repo isn't ready for an agent to work in it — missing context files, raw credentials in agent-visible places, floating MCP server versions, no way to verify the agent's changes. Charter measures that readiness and tells you what to fix.

## What Charter checks

Charter evaluates every repo on three axes:

- **Context** — Can the agent understand where it is and what rules apply? Charter checks that an agent context file exists, stays current, declares edit boundaries, and fits within a context window.
- **Safety** — Is the repo configured so agents can operate without creating obvious security or supply-chain risk? Charter checks for raw secrets in agent-visible files, unpinned or deprecated MCP servers, unreviewed remote origins, and dangerous hook commands.
- **Operability** — Can the agent run the project and verify its own work? Charter checks for automated tests, a discoverable verification command, a reproducible toolchain, and a CI gate.

Governance cuts across all three: suppression entries are audited for reason, ownership, and accumulation rate.

## What Charter is not

Charter is not a code reviewer, a style linter, a package vulnerability scanner, a runtime agent monitor, or a secrets vault. Those tools exist. Charter focuses on one narrower question: is this repo safe and legible for agent-assisted work?

## How it works

`charter doctor` scans the repo statically — no network calls, no LLM calls, no data sent anywhere. It produces a score from 0–100 using a public formula, emits structured findings with rule IDs and remediation guidance, and exits non-zero when the score falls below a threshold you control.

`charter fix` applies safe diff-first repairs for findings Charter knows how to fix. Every repair shows a unified diff before writing. Secrets and dangerous commands are never auto-fixed.

The score, the formula, and the rule set are all public and stable within a major version. Teams can use the score as a local signal, a pre-commit gate, and a CI requirement — and the number means the same thing in all three contexts.

## Who uses Charter

- **Developers** adopting coding agents in an existing repo — to baseline the current state and close the obvious gaps before trusting agents with real tasks.
- **Platform and security engineers** — to gate pull requests on agent-readiness and enforce a consistent policy across repos.
- **Maintainers and reviewers** — to get a deterministic, explainable signal before letting agents edit code in a shared codebase.
