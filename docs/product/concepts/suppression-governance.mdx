---
title: "Suppression Governance"
description: "How Charter treats suppressions as governed exceptions instead of silent muting."
---

Charter allows suppressions because real repositories have false positives, accepted risk, and migration windows.

But Charter does **not** treat suppressions as free-form muting. A suppression changes the trust model of the scan, so it is governed like any other exception path.

## Why suppressions are governed

An unbounded suppression mechanism creates two problems fast:

- teams stop fixing issues because muting is easier than review
- later reviewers cannot tell which risks were understood versus forgotten

Charter's governance model is built to prevent both.

It does that by requiring suppressions to be:

- visible in source control
- attributable with a reason
- time-bounded by default
- reviewable when made permanent

## Two suppression surfaces

Charter supports two ways to record a suppression.

### File-based suppressions

`.charter-suppress.yml` is the repo-level governance file.

It is the right choice when:

- the finding has no precise line location
- the target file cannot carry inline comments
- the team wants a central, reviewable suppression ledger

Typical fields:

- `rule`
- `reason`
- `expires`
- `approver`
- optional `path`

### Inline suppressions

Inline suppressions use a `charter:ignore` directive on the finding's own line.

Example:

```text
# charter:ignore AE-MCP-001 reason="vendored test fixture"
```

Charter only discovers inline suppressions at a finding location. It does **not** scan the whole repo for comments that merely mention the syntax. That keeps the mechanism precise and prevents documentation pages from being misread as suppressions.

## Default TTL and permanent waivers

Charter's model is time-bounded by default, but that default applies differently depending on how the suppression was created.

When you use `charter suppress`, the default is `--expires 90d`, and the command stores that as an absolute `YYYY-MM-DD` date in `.charter-suppress.yml`.

That behavior applies to **CLI-generated suppressions**.

For **hand-authored suppressions**, a missing `expires` field is still honored. Charter does not infer a creation-time 90-day window for a bare entry.

That gives teams a clean default behavior:

- CLI-generated short-term suppressions are easy to add
- the finding re-surfaces automatically later
- old risk acceptance does not stay hidden forever by accident

Permanent waivers are treated differently.

### Default-TTL suppressions

A suppression with no explicit `expires: permanent` is **not** treated as a permanent waiver.

That means:

- finite expiries are honored until they expire
- the CLI's default 90-day path keeps **generated** suppressions bounded
- a hand-authored entry with no `expires` is still honored and is **not** automatically bounded by when it was created
- expired or malformed dates fail closed and do not suppress anything

### Permanent suppressions

Only `expires: permanent` is a permanent waiver.

And permanent waivers are honored **only** when they include an `approver`.

Without an approver:

- Charter flags `AE-SUPPRESS-002`
- the underlying finding stays active
- the accepted risk is treated as undocumented

This rule applies even to secret findings. Charter does not create a special non-suppressible class; it requires stronger governance instead.

## Every suppression needs a reason

`AE-SUPPRESS-001` enforces the simplest rule in the model: every suppression must explain itself.

A good reason answers:

- what is being accepted
- why that acceptance is safe or necessary
- what context a future reviewer would otherwise have to rediscover

"False positive" by itself is usually not enough. "Vendored test fixture" or "lives in a separate infra repo" is much better.

## Governance rules are part of every scan

Charter audits suppressions on every run through three governance rules:

| Rule | What it catches |
|---|---|
| `AE-SUPPRESS-001` | Missing `reason` |
| `AE-SUPPRESS-002` | Permanent waiver without `approver` |
| `AE-SUPPRESS-003` | High suppression rate |

`AE-SUPPRESS-003` is informational only. It exists to surface a repo-level smell: if a large share of findings are suppressed, the team may have stopped trusting the scanner or accumulated too much accepted risk.

The threshold is fixed in v1 at more than 30% suppressed findings.

## What happens to the score

Charter partitions findings before scoring.

That means:

- active findings affect the score
- honored suppressions are listed separately and excluded from the base score
- informational governance findings do not deduct points

This is why suppression governance matters so much: the mechanism can change the final score, so the repo needs an audit trail for using it.

## Practical guidance

Use suppression when the finding is real but temporarily accepted, or when the rule is correct in general but wrong for a narrow, documented case.

Do **not** use suppression as a substitute for fixing the issue when the remediation is straightforward.

In practice, a healthy suppression flow looks like this:

1. confirm the finding really is accepted or noisy in this repo
2. add a meaningful `reason`
3. use a finite expiry unless permanence is truly justified
4. add an `approver` for permanent waivers
5. let the governance rules keep the exception visible over time
